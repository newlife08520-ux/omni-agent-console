# 查單中僅回「商品名」（如天鷹包）卻被轉人工 — 觸發點完整說明

## 情境還原

- 用戶：「我要查別筆」
- AI：「請提供另一筆的訂單編號，或商品名稱＋下單手機」
- 用戶：「**天鷹包**」（只給商品名，尚未給手機）
- 系統卻送出：「目前人工客服暫時忙碌中，已幫您排入待處理清單…」＋「這邊先幫您轉接真人專員處理，請稍後。」

亦即：**用戶正在依引導補資料（查別筆 → 商品名），沒有要求轉真人，卻被轉人工。**

---

## 一、兩則訊息的來源

| 使用者看到的句子 | 程式來源 | 何時出現 |
|------------------|----------|----------|
| 「目前人工客服暫時忙碌中，已幫您排入待處理清單，上班後會依序回覆。」 | `getTransferUnavailableSystemMessage(reason)`（`routes.ts` 約 L442–L447） | 進入「轉人工」流程且 `assignment.assignCase()` 無可用客服（`getUnavailableReason()` 為 `all_paused` 或 null）時，寫一筆 **system** 訊息。 |
| 「這邊先幫您轉接真人專員處理，請稍後。」 | `HANDOFF_MANDATORY_OPENING` → `getHandoffReplyForCustomer(...)`（`phase2-output.ts`） | 任一「轉人工」分支在送給客戶的 **ai** 訊息中使用。 |

因此：**只要任一「轉人工」觸發點被觸發，且當時無可用客服，就會出現「忙碌中」＋「轉接真人專員」這兩句。**

---

## 二、可能導致此情境的觸發點（由前到後）

以下為 `autoReplyWithAI` 內實際會執行到的轉人工路徑，依程式執行順序排列。

### 觸發點 1：Webhook 關鍵字（一進線就轉）

- **位置**：LINE Webhook 收訊後、進 `autoReplyWithAI` 之前（`routes.ts` 約 L4805–L4814）。
- **條件**：本則用戶訊息 **包含** 後台 `human_transfer_keywords` 的任一句（預設：「我要轉人工」「轉人工」「找真人客服」「找主管」）。
- **本情境**：「天鷹包」不包含上述關鍵字 → **不會**由此觸發。

---

### 觸發點 2：高風險短路（法務／公關）

- **位置**：`routes.ts` 約 L3782–L3816。
- **條件**：`state.customer_emotion === "high_risk"` 或 `riskCheck.reasons` 命中（如詐騙、檢舉、投訴、消保官、公開、發文、再不處理等）。
- **本情境**：「天鷹包」不會被判為高風險 → **不會**由此觸發。

---

### 觸發點 3：尷尬／重複轉人工（**最可能**）

- **位置**：`routes.ts` 約 L3947–L4002，`shouldHandoffDueToAwkwardOrRepeat()`。
- **條件**：以下任一條成立即 `shouldHandoff === true`，且**未**被「查單主線收斂」排除：
  1. **sameDataAskedTwice**：近期 AI 訊息中，符合「討訂單/手機/商品」句型（`ASK_ORDER_PHONE_PATTERNS`）的 **≥ 3 則**。
  2. **sameTemplateRepeatedTwice**：**最近兩則 AI 回覆**內容高度相似（完全相同或前 50 字相同）。
  3. **userSaidAlreadyGaveAndLastAiAskedAgain**：用戶訊息符合「我給過了／你沒看／前面有…」，且上一則 AI 在討訂單/手機，且更早的用戶訊息裡有訂單編號或手機。
  4. **intentMismatchLastRound**：本輪意圖為查單、上一則用戶是查單相關、上一則 AI 卻回表單／退換貨（非查單）。

**為何「天鷹包」容易在這裡觸發：**

- 對話通常為：  
  - 第一次查單：AI「請提供訂單編號或商品+手機…」  
  - 用戶：「我要查別筆」  
  - AI  again：「請提供另一筆的訂單編號或商品名稱+手機…」  
- 上述兩則 AI 內容前 50 字常幾乎一樣 → **sameTemplateRepeatedTwice** 為 true。
- 此時用戶回「天鷹包」是在**依指示補「商品名稱」**，但尷尬檢查**不區分**「用戶正在補資料」或「用戶在抱怨重複」；只要「最近兩則 AI 很像」就轉人工。
- 「查單主線收斂」的排除條件（`skipAwkwardHandoffDueToActiveOrder`）只放行「已有當前訂單＋本輪是訂單延伸問句」（如什麼時候到貨、好的、謝謝等）；**「天鷹包」是補查別筆的資料，不符合該延伸句型**，因此不會被排除。

→ **結論**：在「我要查別筆 → AI 再問一次訂單/商品+手機 → 用戶回天鷹包」的流程下，最常見的就是 **觸發點 3（sameTemplateRepeatedTwice）**，導致轉人工＋「忙碌中」＋「轉接真人專員」。

---

### 觸發點 4：state 判定為 handoff（明確要真人）

- **位置**：`reply-plan-builder.ts` L72–L76，`buildReplyPlan()`；`routes.ts` 約 L4005–L4060。
- **條件**：`state.needs_human && state.human_reason` 為真，且 `human_reason` 為 explicit_human_request / legal_or_reputation_threat / payment_or_order_risk / policy_exception / return_stage_3_insist / repeat_unresolved 等。
- **本情境**：「天鷹包」會被 `detectPrimaryIntent` 判為 **unclear**（無訂單/查單/退貨等關鍵字，長度 3 也未達 5），且 **unclear** 在 `AI_HANDLABLE_INTENTS` 中會把 `needs_human` 清掉，故不會設為 handoff → **不會**由此觸發。

---

### 觸發點 5：「已給過」但三層搜尋找不到

- **位置**：`routes.ts` 約 L4190–L4233。
- **條件**：用戶訊息符合 `isAlreadyProvidedMessage`（我給過了／你拿過了／前面有／已經給過…），且 `searchOrderInfoThreeLayers` 在近期訊息＋圖片＋linked order 都**找不到**訂單編號或手機。
- **本情境**：「天鷹包」不是「我給過了」這類句子 → **不會**由此觸發。

---

### 觸發點 6：LLM 呼叫 `transfer_to_human` 工具

- **位置**：`routes.ts` 約 L4393–L4418、L5475–L5492；工具執行後會設 `needs_human=1` 並送 handoff 句。
- **條件**：模型在回覆時主動呼叫 `transfer_to_human`（例如誤判用戶要轉真人、或不知如何處理就轉）。
- **本情境**：若前面**沒有**被觸發點 3 短路掉，理論上會進 LLM；有機會因 prompt/情境被模型呼叫轉人工，但實務上在「天鷹包」情境多數會先被觸發點 3 攔截。

## 三、流程簡圖（簡化）

```
用戶送「天鷹包」
  → resolveConversationState → primary_intent = "unclear"
  → buildReplyPlan → mode = "answer_directly"（非 handoff）
  → shouldHandoffDueToAwkwardOrRepeat(recentMessages)
       → sameTemplateRepeatedTwice(recentMessages) === true
         （最近兩則 AI 皆為「請提供訂單編號或商品+手機…」）
  → awkwardCheck.shouldHandoff === true
  → skipAwkwardHandoffDueToActiveOrder === false（「天鷹包」非訂單延伸句）
  → 進入尷尬轉人工分支
  → assignment.assignCase() 無可用客服
  → 寫 system 訊息：getTransferUnavailableSystemMessage() → 「目前人工客服暫時忙碌中…」
  → 送 ai 訊息：getHandoffReplyForCustomer(...) → 「這邊先幫您轉接真人專員處理，請稍後。」
```

## 五、建議修正方向（簡要）

1. **查單脈絡下「補資料」不觸發 sameTemplateRepeatedTwice**  
   - 當 **上一則 AI 在引導查單**（內容符合討訂單/商品+手機）且 **本輪用戶訊息像是在補資料**（例如：僅商品名、僅訂單編號、僅手機、或「天鷹包」「DEN65234」等短句）時，**不要**因「最近兩則 AI 很像」就轉人工，可視為「用戶正在依指示補齊查別筆資料」。
2. **放寬 skipAwkwardHandoffDueToActiveOrder**  
   - 在「查單脈絡」且「本輪為補訂單編號／商品名／手機」時，也視為可放行、不因尷尬重複轉人工（或新增一項「查單補資料」的 bypass 條件）。
3. **sameTemplateRepeatedTwice 門檻或情境**  
   - 可考慮：僅在「同一輪／同一主題」已問過 **≥ 3 次**類似句型時才觸發；或排除「上一則 AI 為查單引導、本則用戶為疑似訂單編號/商品名/手機」的狀況。

以上三點可單獨或搭配實作，以減少「查單中只回商品名卻被轉人工」的發生率。
