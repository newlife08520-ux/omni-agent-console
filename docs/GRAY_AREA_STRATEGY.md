# Gray Area 長期處理策略（設計）

## 現狀
- 灰區留言僅標記 `main_status = gray_area`，不執行回覆／隱藏／導 LINE。
- 灰區會出現在「例外優先」列表，方便人工檢視。

## 未來預計處理方式

### 1. 自動歸檔
- **建議**：灰區留言在 N 天（例如 3～7 天）內未有人工處理，可自動改為 `completed` 或專用狀態 `gray_archived`，並從例外列表排除。
- **實作**：排程或背景 job 依 `created_at` + `main_status = 'gray_area'` 判斷，超過閾值則更新狀態。
- **好處**：避免灰區無限堆積，列表維持可管理。

### 2. 灰區抽查
- **建議**：提供「灰區抽查」按鈕（類似「抽查已完成」），隨機列出約 20 筆灰區留言，供人工抽樣檢查是否有誤判或需升級處理。
- **實作**：API `GET /api/meta-comments?status=gray_area&random=20` 或專用 `/api/meta-comments/spot-check-gray`。
- **好處**：不預設全列灰區，但可主動抽樣優化規則。

### 3. 部分 gray_area 規則允許自動回覆
- **建議**：在規則桶 `gray_area` 的規則上新增選項，例如 `action_reply_allowed_in_gray = 1`；若最終採用為灰區且該規則允許，則仍可送出簡短回覆（不隱藏、不導 LINE）。
- **實作**：在 `meta_comment_risk_rules` 加欄位（或沿用現有 action_reply），在 auto-execute 中對 gray_area 分支再判斷是否執行 reply。
- **好處**：例如「哈哈」「好喔」類可自動回一句簡短話，其餘維持僅標記。

### 4. 灰區升級
- **建議**：人工在詳情頁可將灰區「升級為待人工」或「標記為已檢視／無需處理」，寫回 `main_status` 或專用欄位。
- **實作**：已有「標記已檢視（灰區）」按鈕，目前**暫時**將 `gray_area` 改為 `completed`（簡化策略，利於快速消化）。
- **未來可拆**：若營運需要區分「已檢視但保留紀錄」與「已檢視並結案」，可新增狀態如 `gray_reviewed`、`gray_completed`，或專用欄位記錄檢視時間，本輪不改結構。

## 建議實作順序
1. 灰區抽查（低成本、立刻有用）
2. 自動歸檔（避免堆積）
3. 灰區規則可選自動回覆（視需求）
4. 灰區升級按鈕（視營運需求）
