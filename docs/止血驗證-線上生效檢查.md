# 止血驗證：線上生效檢查

## 1. 後台 human_transfer_keywords 已強制縮為四項

- **程式**：每次 server 啟動時會跑 migration `migrateBleedHumanTransferKeywords()`，將 `settings.human_transfer_keywords` 強制設為：  
  `我要轉人工,轉人工,找真人客服,找主管`
- **不需手動改後台**：重啟後 DB 即為上述四項。若你曾手動改過，重啟後也會被覆寫成這四項。
- **若要確認**：登入後台 → 系統設定 → 智能轉人工觸發設定，應只看到這四個關鍵字。

## 2. 確認線上 deploy 與重啟時間

- **版本與啟動時間 API**（不需登入）：
  ```bash
  curl -s https://你的網域/api/version
  ```
  回傳範例：
  ```json
  {
    "version": "1.0.0",
    "commit": "abc1234",
    "startTime": "2026-03-12T08:00:00.000Z"
  }
  ```
- **建議**：在 CI 建置時設定 `COMMIT_SHA`、`VERSION` 環境變數，部署後呼叫 `/api/version` 即可確認目前跑的是哪個 commit、何時啟動。
- **確認 routes / conversation-state-resolver / awkward-repeat-handoff 是否生效**：  
  同上，部署的是同一份 codebase；只要 `/api/version` 的 commit 與你預期一致，這三處邏輯即為該版本。

## 3. 線上實測止血 5 題

- **五句**：`在嗎`、`人呢`、`太誇張了`、`很煩`、`你有沒有看`
- **預期**：這五句都**不可以**  
  - 直接切待人工接手  
  - 直接 AI 靜音  
  - 直接送轉真人句  

**做法**：  
在 LINE（或你串的渠道）對客服帳號依序送這五則訊息，然後在後台確認：

- 該聯絡人**未**變成「待人工」、**未**被標成 needs_human、**未**收到「轉接真人專員」那則固定句。  
- 若可以看 DB：該 contact 的 `needs_human` 應為 0（且本輪沒有因這五句而被設成 1）。

## 4. 若五題仍有任一題 needs_human=true，查 log 確認來源

程式已在下列時機寫入 **needs_human=1** 的來源 log，方便查是哪一層設進去的：

| 來源 | Log 關鍵字 |
|------|------------|
| Webhook 關鍵字（LINE 文字） | `[WEBHOOK] needs_human=1 source=webhook_keyword msg=` |
| Webhook 關鍵字（FB 文字） | `[FB WEBHOOK] needs_human=1 source=webhook_keyword msg=` |
| State resolver（本輪意圖／法務風險等） | `[Webhook AI] needs_human=1 source=state_resolver reason=` |
| 尷尬／重複轉人工 | `[Webhook AI] needs_human=1 source=awkward_repeat reason=` |
| 已給過但找不到 → 轉人工 | `[Webhook AI] needs_human=1 source=already_provided_not_found msg=` |
| 高風險短路（法務／公關） | `[Webhook AI] needs_human=1 source=high_risk_short_circuit reasons=` |

**查法**（依你實際 log 方式調整）：

```bash
# 例：看最近是否有因「關鍵字」設 needs_human
grep "needs_human=1 source=webhook_keyword" /path/to/logs

# 例：看是否為 state_resolver（意圖／human_reason）
grep "needs_human=1 source=state_resolver" /path/to/logs

# 例：看是否為尷尬重複
grep "needs_human=1 source=awkward_repeat" /path/to/logs
```

若這五句仍觸發 handoff，對照上述 log 即可判斷是：  
**webhook_keyword**、**state_resolver**、**awkward_repeat**、**already_provided_not_found**、**high_risk_short_circuit**，或**舊 contact 狀態殘留**（例如前幾輪已 needs_human=1，本輪未清掉且又送 post handoff 句；此時可再確認是否已套用「本輪 AI 可處理意圖不補 handoff 句」邏輯）。

## 5. 在機器上跑驗證腳本（與線上同邏輯）

在**與 production 同一台機器或同一 DATA_DIR** 執行，會用同一份 DB 設定跑止血 5 題邏輯並列印結果：

```bash
cd /path/to/Omni-Agent-Console
npx tsx server/scripts/verify-handoff-bleed.ts
```

輸出會包含：每句是否關鍵字命中、意圖、needs_human、awkward、plan.mode、以及若會 handoff 時的**來源**。  
若五題皆不 handoff，腳本會 [PASS] 並 exit 0；若有任一句會觸發，會 [FAIL] 並列出來源後 exit 1。
