# 為什麼不會自動回覆 — 排查步驟

客戶傳訊息到 LINE/FB 後，**訊息有進中控台**，但**客戶端收不到 AI 自動回覆**時，請依下列順序檢查。

---

## 流程簡述

1. **Webhook** 收到訊息 → 寫入 DB、廣播 SSE → 若該渠道 **AI 已啟用** 且 **非轉人工關鍵字**，會呼叫 `enqueueDebouncedAiReply` 把「要回覆」的任務放進 **Redis 佇列**。
2. **AI Reply Worker**（獨立進程）從 Redis 取出任務 → 呼叫主站的 **`POST /internal/run-ai-reply`** → 主站跑 AI、寫入回覆、用 LINE/FB API **推訊息給客戶**。
3. 若 **Worker 沒在跑**，任務會一直留在佇列裡，**不會有人執行**，客戶就不會收到自動回覆。

---

## 1. 確認有排進佇列（看主站 log）

發一則**一般文字**（不要觸發「轉人工」關鍵字）到 LINE 後，看 **Railway Deploy Log** 或本機終端：

- 應出現：`[WEBHOOK] AI 已啟用，準備文字自動回覆`
- 接著：`[Queue] debounced job scheduled: ai-reply:line: <聯絡人ID>`

若看到 **`[WEBHOOK] AI 已關閉 ... 跳過自動回覆`** → 到 **系統設定 → 品牌 → 該品牌 → LINE 渠道**，確認 **「AI 自動回覆」** 已開啟。

若看到 **`needs_human=1`** → 該則訊息被判定為要轉真人，不會走 AI 回覆（會改發轉人工開場白）。

若**完全沒有** `debounced job scheduled`，且也沒有「AI 已關閉」→ 檢查 **REDIS_URL** 是否已設；未設時不會使用佇列，會走舊路徑（需同進程有 worker）。

---

## 2. 確認 Worker 有在跑（最常見原因）

**AI 自動回覆是由「獨立的 Worker 進程」執行的，不是網頁主站。**

- **本機**：除了 `npm run dev`（或 `npm run start`）外，**還要再開一個終端**執行：  
  **`npm run dev:worker`**（開發）或 **`npm run start:worker`**（生產，需先 `npm run build`）。
- **Railway**：預設只會跑一個指令（例如 `npm start`），**不會自動跑 Worker**。  
  要自動回覆，必須**再開一個 Service**，啟動指令設為 **`npm run start:worker`**（且 build 階段需有 `npm run build` 產生 `dist/`）。

若 Worker **沒跑**：
- 主站 log 仍會出現 `[Queue] debounced job scheduled: ai-reply:line: xxx`
- 但 **沒有人從 Redis 取 job**，所以不會呼叫 `/internal/run-ai-reply`，客戶就**不會收到任何回覆**。

**Railway 建議作法**：同一個專案下新增一個 **Worker Service**，與主站共用同一份程式碼與環境變數，**Build 指令**與主站相同（例如 `npm run build`），**Start 指令**改為 `npm run start:worker`。

---

## 3. Worker 環境變數（Railway 第二個 Service）

Worker 進程需要以下環境變數（在 Railway Worker Service 的 Variables 設定，可與主站共用部分變數）：

| 變數 | 必填 | 說明 |
|------|------|------|
| `REDIS_URL` | ✅ | 與主站相同，Worker 靠它取佇列任務 |
| `INTERNAL_API_SECRET` | ✅ | 與主站**完全一致**，呼叫 `/internal/run-ai-reply` 時帶在 header，主站會驗證 |
| `INTERNAL_API_URL` | 建議設 | Worker 要打的主站網址。**同機**可設 `http://localhost:8080`（port 與主站一致）；**Railway 上 Worker 與主站為不同 Service** 時，設主站對外網址，例如 `https://你的主站.up.railway.app` |

主站（API）也必須設定 **`INTERNAL_API_SECRET`**，且與 Worker 的 **值相同**，否則 Worker 打 `/internal/run-ai-reply` 會回 401/403，回覆仍不會送出。

---

## 4. 若仍沒回覆：看 Worker 與主站 log

- **Worker log**：應出現 `[Worker] processing: ...`、`[Worker] sent: ...`。若出現 `internal/run-ai-reply 401` 或 `403` → 檢查 **INTERNAL_API_SECRET** 是否與主站一致；若出現 `fetch failed` / `ECONNREFUSED` → 檢查 **INTERNAL_API_URL** 是否正確（Worker 能否連到主站）。
- **主站 log**：收到 Worker 請求時會處理 AI 並呼叫 LINE/FB API。若主站有錯誤（例如 LINE Token 過期、API 限流），也會導致客戶收不到回覆，請依主站錯誤訊息排查。

---

## 快速檢查表

| 項目 | 如何確認 |
|------|----------|
| 有排進佇列 | 主站 log 有 `[Queue] debounced job scheduled: ai-reply:line: xxx` |
| 渠道 AI 已開 | 主站 log 有 `[WEBHOOK] AI 已啟用，準備文字自動回覆` |
| Worker 有跑 | 本機有開 `npm run dev:worker` / Railway 有第二個 Service 跑 `start:worker` |
| Worker 能連主站 | Worker 設好 INTERNAL_API_URL、INTERNAL_API_SECRET，主站也有設 INTERNAL_API_SECRET 且一致 |
| 主站有 INTERNAL_API_SECRET | Railway 主站 Variables 有設，與 Worker 相同 |

**最常見原因**：**只部署了主站，沒有跑 Worker**，所以佇列有任務但沒人執行，就不會自動回覆。請先確認本機或 Railway 上是否有執行 `start:worker`。
