# 客服流程架構檢視與衝突點（修改前 Review）

## 一、現有架構摘要

### 1. 訊息處理入口（routes.ts）

- **LINE**: `POST /api/webhook/line` → 解析事件 → `debounceTextMessage` → `withContactLock` → `autoReplyWithAI(contact, mergedText, token, brandId)`
- **FB Messenger**: `POST /api/webhook/facebook` → `entry.messaging` → 同上 `debounceTextMessage` → `autoReplyWithAI(..., "messenger")`

### 2. autoReplyWithAI 現有流程（單一入口、無 state / plan 分層）

1. **前置檢查**：`awaiting_human` / `high_risk` / `needs_human` / AI 靜音 → 直接 return，不進 LLM  
2. **高風險關鍵字**：`detectHighRisk(userMessage)` → 若命中，設 high_risk + needs_human，寫 system message + AiLog，return  
3. **安全確認分流**：`classifyMessageForSafeAfterSale(userMessage)`（詐騙/他平台/待確認）→ 若命中，回模板 + 可設 suggest_human，return  
4. **圖片＋極短文字**：`isShortOrAmbiguousImageCaption` + `shouldEscalateImageSupplement` → 回補充模板或升級人工，return  
5. **標記與更新**：`detectReturnRefund` → 寫 issue_type；`detectIssueType`、`detectIntentLevel`、`classifyOrderNumber`、`suggestTagsFromContent`、`computeCasePriority` → 更新 contact  
6. **組 prompt**：`getEnrichedSystemPrompt(brandId)` = 全域 system_prompt + 品牌 system_prompt + **handoffBlock（含 returnFormUrl）** + 人工時段 + 商品目錄 + 知識庫 + 圖片素材  
7. **呼叫 OpenAI**：同一輪可多次 tool call（order lookup、transfer_to_human、send_image），**沒有「本輪只走一條主流程」的約束**  
8. **transfer_to_human**：工具被呼叫時寫入 needs_human、awaiting_human、assignment、system message、alert  
9. **多次查單失敗**：`orderLookupFailed >= 2` 且未轉人工 → 自動轉人工  

**沒有**：conversation state 結構、reply plan、24h 閒置結案、評價 eligibility 條件、QA 規則打分。

### 3. Prompt 現狀

- **全域**：`storage.getSetting("system_prompt")`，預設「你是一位專業的客服助理。」（DB 預設值內含訂單查詢決策樹等長文，見 db.ts 種子）  
- **品牌**：`brand.system_prompt`，整段塞進「品牌專屬指令」  
- **Policy / Handoff**：全部寫在 `handoffBlock` 字串（F2、轉人工五種、退換貨三階段、禁止事項、表單 URL）  
- **Router**：無獨立 router prompt；意圖與流程完全由 LLM 自由判斷 + 工具選擇  

**衝突點**：  
- 退換貨「先表單」與「先查單／先分平台」在 prompt 裡並存，LLM 容易同輪既查單又轉人工或先講平台。  
- 沒有「本輪唯一 mode」，容易多流程搶答。

### 4. Handoff 規則（目前全在 prompt + 工具描述）

- 轉人工條件：五種 reason（explicit_human_request、legal_or_reputation_threat、payment_or_order_risk、policy_exception、repeat_unresolved、return_stage_3_insist）  
- F2：第一句禁止區分官方/其他平台；僅在「已提供資料 + 已查詢 + 查無」才可提其他通路  
- 實作上：**沒有**後端 state（如 return_stage、needs_human 由 state 驅動），needs_human 只由關鍵字或 tool call 寫入；**沒有**「退換貨首輪強制 return_form_first」的程式邏輯。

### 5. 訂單查詢

- 工具：`lookup_order_by_id`、`lookup_order_by_product_and_phone`（+ 日期+個資等）  
- 觸發：完全由 LLM 決定是否呼叫；沒有「先判斷 primary_intent 再決定是否開放查單」  
- **衝突**：客人說「我想退貨，MRQ55276」時，LLM 可能先走查單再回退貨，與「退換貨先表單、不先查平台」打架。

### 6. 表單導流（return_form_url）

- **來源**：`brand.return_form_url` 或預設 URL，塞進 handoffBlock 的「售後表單：${returnFormUrl}」  
- **行為**：僅在 prompt 中「合適時機提供」，**沒有**後端強制「refund/return 首輪 mode = return_form_first」或「本輪禁止查單」的邏輯。

### 7. 閒置／結案

- **現狀**：**沒有** 24 小時閒置結案 job。  
- 結案僅在：後台手動 PUT status = resolved/closed；`assignment.closeCase` 會寫 `closed_at`、`closed_by_agent_id`。  
- **衝突**：需求為「客戶 24h 未回覆一定結案」，目前無任何定時任務做此事。

### 8. 評價邀請

- **觸發**：僅在 **手動** PUT `/api/contacts/:id` 且 `status === "resolved" | "closed"` 時，若 `needs_human===1` 且 `cs_rating==null` 發真人評價；否則若 `ai_rating==null` 發 AI 評價。  
- **沒有**：resolution_status、rating_eligible、rating_invited_at、未解決/退貨爭議/已轉人工不發等條件。  
- **衝突**：需求為「僅在 resolved/closed + 非投訴/非退貨爭議/非待人工 + 未發過」才發，目前是「一結案就發」，沒有 eligibility 檢查。

### 9. 聯絡人／Contact 狀態欄位（schema + DB）

- **已有**：status、needs_human、closed_at、closed_by_agent_id、issue_type、last_message_at、tags、intent_level、order_number_type、case_priority、assignment_status…  
- **缺少**（需求要求）：resolution_status、waiting_for_customer、human_reason、return_stage、rating_invited_at、close_reason、qa_score、qa_score_reason  

### 10. 後台設定

- **已有**：品牌有 return_form_url；全域 system_prompt；superlanding / SHOPLINE 金鑰  
- **缺少**：idle_close_hours、rating_enabled、rating_delay_minutes、afterhours_handoff_notice_enabled 等集中設定（目前僅分散在程式或未實作）

## 二、衝突點整理

| 項目 | 現狀 | 目標 | 衝突 |
|------|------|------|------|
| 流程決策 | LLM 自由選工具與內容 | 先 state → plan → 再生成 | 同輪多流程搶答、退貨 vs 查單 vs 通路判斷打架 |
| 退換貨 | Prompt 說三階段+表單時機 | 退換貨**優先**導表單，首輪不查單、不分平台 | 沒有「return_form_first」強制、F2 僅在 prompt |
| F2 | 僅寫在 handoffBlock | 後端硬規則，最高優先 | 需 state/plan 層擋住「首輪提其他平台」 |
| 24h 結案 | 無 | 24h 未回必結案，分流結語 | 需 idle-close job + 結案原因/分流 |
| 評價 | 結案即發 | 僅 resolved/closed + 非爭議 + 未發過 | 需 rating_eligibility 模組 + 欄位 |
| 高風險/已轉人工 | 可被手動關閉 | 禁止 auto close | 需在 idle-close 排除 |
| 分數 | 無 | 規則 QA 扣分 | 需 qa-score-calculator |
| Prompt | 一大塊 | persona / policy / router / brand 拆分 | 需重構 getEnrichedSystemPrompt 與 router 固定模板 |

## 三、這次一定要改的後端項目（對應需求一）

1. **對話 state resolver**（conversation-state-resolver.ts）  
   - 輸入：contact、最近訊息、訂單查詢結果標記  
   - 輸出：ConversationState（primary_intent、needs_human、human_reason、return_stage、order_info_status、waiting_for_customer、resolution_status、rating_eligible、last_customer_reply_at 等）

2. **Reply plan builder**（reply-plan-builder.ts）  
   - 輸入：ConversationState、設定（idle_close_hours、return_form_url）  
   - 輸出：ReplyPlan（mode 單一、must_not_include / should_include）  
   - **流程優先級**：明確要真人 > 法務/投訴 > 金流/訂單爭議 > 明確堅持退款退貨 > 退貨/退款/取消/久候 > 訂單查詢/出貨 > 商品諮詢 > 閒聊/模糊

3. **退換貨優先導表單 + F2 硬規則**  
   - 當 primary_intent 為 refund_or_return / exchange_request / cancellation_request 且非超高風險：plan.mode = return_form_first（或 return_stage_1），**不**走 order_lookup；  
   - F2：在 state 或 plan 層禁止「首輪出現官方/其他平台」說法；reply generator 的 policy 保留 F2 文案。

4. **24 小時 idle close job**（idle-close-job.ts）  
   - 定時掃 last_message_at 為客戶最後一則且超過 24h 的 contact；  
   - 排除：已轉人工（awaiting_human）、高風險（high_risk）；  
   - 分流結語（一般諮詢 / 待補單號 / 退換貨待填表 / 已轉人工不關閉）；  
   - 寫 closed_at、close_reason、status=closed。

5. **Rating eligibility**（rating-eligibility.ts）  
   - 條件：resolution_status in resolved|closed、非投訴/法務/高風險、非退貨爭議未完、非已轉人工待處理、客戶語氣非 angry/high_risk、尚未發過（rating_invited_at 或 ai_rating/cs_rating 未填）；  
   - 僅在通過時才允許發送評價邀請（含自動結案後發送）。

6. **QA 規則打分**（qa-score-calculator.ts）  
   - 從 100 分起算，違反 F2、該轉未轉、不該轉卻轉、知識亂答、未解決發評價、同輪多事、重複問資料 等扣分；  
   - qa_score 由系統算，qa_score_reason 可 LLM 摘要。

7. **高風險與已轉人工禁止 auto close**  
   - 在 idle-close-job 中明確排除 status in (awaiting_human, high_risk)；已轉人工 24h 未接可走內部告警，不對客發結案語。

8. **避免多流程同輪搶答**  
   - Reply plan 只產一個 mode；reply generator 依 plan + persona + policy + knowledge 生成，不讓 LLM 再自行選流程。

## 四、建議修改順序（實作時）

1. **DB/schema**：新增 resolution_status、waiting_for_customer、human_reason、return_stage、rating_invited_at、close_reason、qa_score、qa_score_reason（及必要索引）。  
2. **conversation-state-resolver.ts**：產出 ConversationState。  
3. **reply-plan-builder.ts**：依 state + 設定產 ReplyPlan，落實流程優先級與 return_form_first。  
4. **routes.ts autoReplyWithAI**：先跑 state resolver → plan builder；若 plan.mode 為 handoff / return_form_first / idle_closure / invite_rating 等，走對應分支（固定文案或限制 LLM 的 system 指令）；其餘再組 prompt 呼叫 LLM（prompt 拆成 persona / policy / router / brand）。  
5. **idle-close-job.ts**：定時掃描、分流結語、寫入結案。  
6. **rating-eligibility.ts**：在發送評價前檢查；結案後若 eligible 可發送。  
7. **qa-score-calculator.ts**：在適當時機（例如 AI 回覆後或結案時）計算並寫入。  
8. **設定**：至少支援 return_form_url（已有）、idle_close_hours、rating_enabled、rating_delay_minutes（config/DB 預設值）；後台 UI 可後補。

## 五、驗收對應（對應需求十三）

- 退貨首輪：不講平台、不先查單、不先轉人工；先承接再導表單。  
- 退貨+單號：不先走查單主流程；仍先承接退換貨、優先表單。  
- 久候不想等：不先懷疑通路；先承接情緒再導表單。  
- 商品尺寸：先確認哪一款；知識庫無則不亂答。  
- 查單：高機率訂單號直接查。  
- 待客補資料 24h：自動結語、結案、可 reopen。  
- 退換貨待填表 24h：表單型結語、結案、可 reopen。  
- 已轉人工：不自動對客結案；保留 awaiting_human；內部告警。  
- 評價：僅在已處理完成可發；未解決售後/退貨不發。  
- 高風險：立即轉人工、不 auto close、不發評價。
