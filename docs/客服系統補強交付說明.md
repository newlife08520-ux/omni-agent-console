## 二、新增的案件狀態流程

| 狀態 (status) | 說明 |
|---------------|------|
| 新案件 (new_case) | 剛建立或剛轉入待處理 |
| 待補資訊 (pending_info) | 需客戶補充分享 |
| 待訂單編號 (pending_order_id) | 系統已要求提供訂單編號，等待客戶提供 |
| 待人工接手 (awaiting_human) | 已轉人工，尚未分配或全員忙碌 |
| 已分配 (assigned) | 已指定負責客服 |
| 處理中 (processing) | 客服正在處理 |
| 等待客戶回覆 (waiting_customer) | 客服已回覆，等待客戶補資料或回覆 |
| 已解決待觀察 (resolved_observe) | 暫結案，觀察是否重開 |
| 已結案 (closed) | 正式結案 |
| 已重開 (reopened) | 結案後客戶又回來 |

**規則摘要**  
- 客服已回覆但客戶未補資料 → 可設為「等待客戶回覆」  
- 系統要求提供訂單編號 → 可設為「待訂單編號」  
- 點轉人工且未被接手 → 「待人工接手」  
- 結案後又來訊 → 可標記「已重開」

## 三、新增的人工分配規則

1. **排除條件**（不列入可接案名單）  
   - 午休中（`lunch_break === 1` 或時段在 lunch_start_time～lunch_end_time）  
   - 下班（`on_duty === 0` 或當前時間不在 work_start_time～work_end_time）  
   - 暫停接案（`pause_new_cases === 1`）

2. **排序**（依序）  
   - 優先順位數字小者優先（priority ASC）  
   - 未結案件數少者優先（open_cases_count ASC）  
   - 今日已分配數少者優先（today_assigned_count ASC）

3. **選人**  
   - 取排序後第一位為下一位接案客服；若無人可接則不分配，案件維持「待人工接手」。

4. **手動改派**  
   - 主管可對已分配案件呼叫改派 API，選擇新客服並寫入 assignment_history。

5. **指派紀錄**  
   - 每筆案件保留：首次分配對象與時間、是否改派、改派紀錄、最終結案人（closed_by_agent_id）。

## 四、新增的優先級排序規則

**案件優先級 (case_priority)：數字愈小愈優先。**

| 優先級 | 條件 |
|--------|------|
| 1 | 標籤為「出貨延遲」「缺貨/欠貨」「退款/取消」「客訴」；或「訂單查詢」+ 高意圖 |
| 2 | 高意圖；或標籤「待人工接手」「高意圖」 |
| 3 | 中意圖；或標籤「待訂單編號」 |
| 4 | 其餘 |
| 5 | 低意圖或標籤「低意圖」 |

**列表排序**（getContacts）  
- 釘選優先 → case_priority 升序（NULL 視為 999）→ last_message_at 降序。

## 五、新增的意圖強度判斷規則

**高意圖 (high)**  
- 明確提到訂單編號、缺貨/欠貨/未出貨/到貨、退款退貨換貨、收到瑕疵、要買但卡住、或出現 KBT/ORD 等訂單格式、5 位以上數字+訂單/單號等。

**中意圖 (medium)**  
- 商品差異/比較、價格/優惠/折扣/活動、何時到貨/補貨、轉人工/轉真人/找客服；或訊息長度 ≥ 8 字且含訂單/出貨/退款等關鍵字。

**低意圖 (low)**  
- 幾乎無文字、僅「好/嗯/謝謝」等單句、或單一數字/選項。

**亂點防呆**  
- 近期多則訊息涵蓋多個不相干主題（如同時出現 欠貨、訂單、優惠、轉人工）→ 降為 medium。

## 六、訂單編號／金流編號辨識規則

**辨識結果 (order_number_type)**  

| 類型 | 條件範例 |
|------|----------|
| order_id | KBT/ORD/MRQ/MRH/DEN/EC/SL + 數字、或 2–4 英文字 + 5 位以上數字 |
| payment_id | T + 10 位以上數字、或內文含「交易序號/金流/付款編號」、或 5 位數字且內文含「末五碼/後五碼」 |
| logistics_id | 10–14 位純數字（非 09 開頭）、或長英數字且內文含「宅配/包裹/物流/黑貓」等 |
| phone | 09 開頭 10 碼、或 0 開頭 10 碼 |
| pending_review | 5 位以上純數字或 6 位以上英數字，無法歸類時 |
| unknown | 其餘 |

**溫和引導**  
- 若為 `payment_id`：回覆「這組看起來比較像付款／金流資訊，不是訂單編號…」並引導至訂單成立信/簡訊/會員訂單頁或提供收件人姓名、手機後三碼。  
- 若為 `logistics_id`：回覆「這組看起來比較像物流單號…」並同樣引導。  
- 後台可依 `order_number_type` 標記「疑似訂單編號／金流編號／物流編號／待人工確認」。

**個人視角（/api/performance/me）**

- 同上，但為「全部客服」列表，每人一筆，含 display_name、agent_id。

## 九、建議截圖清單（3～5 張）

請在實際環境中截圖以下畫面，以利驗收與說明：

1. **即時客服 — 案件列表**  
   - 左側聯絡人列表出現「狀態標籤」與「負責人」（例如：👤 客服小李）。  
   - 若有未讀人工案件，側欄「即時客服」旁出現紅點與數字。

2. **即時客服 — 分配／改派**  
   - 選一筆「待人工接手」案件，標題列出現「分配」按鈕。  
   - 選一筆已分配案件，標題列出現「改派」按鈕與「負責：XXX」。  
   - 點「改派」後出現改派對話框（選擇客服、備註、確認改派）。

3. **客服績效 — 個人**  
   - 以客服身份登入，進入「客服績效」頁。  
   - 畫面顯示：今日新進、待處理、處理中、今日已結案、累計已結案、平均首次回覆時間、平均結案時間、結案率/解決率。

4. **客服績效 — 主管報表**  
   - 以主管身份登入，進入「客服績效」→「主管報表」分頁。  
   - 畫面顯示：今日總案件數、待處理數、轉人工數、各客服處理量、常見標籤排行、問題類別占比。

5. **午休／全員忙碌**  
   - 當全員午休或暫停時，客戶轉人工後對話中出現系統訊息：「目前人工客服暫時忙碌中，已幫您排入待處理清單，上班後會依序回覆。」  
   - 該案件標籤含「午休待處理」（可在案件詳情或列表中確認）。
