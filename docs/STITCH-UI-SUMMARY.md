# Stitch UI — 全站摘要（供設計／生成式改版對齊）

本文件依 `client/src` 實際路由、側欄、頁面主檔整理；產品內無「審判官」字樣，**監控／判讀語意**主要落在 **留言中心（AI 自動處理監控台）** 與 **即時客服三欄工作台**。

---

## A. 全站共通規則

### 主按鈕（Primary CTA）
- **定義**：完成當頁「主任務」的單一步驟，例如：登入、送出對話、立即公開回覆、儲存 Prompt、一鍵建立渠道、確認新增成員。
- **視覺**：實作上多為 `bg-emerald-600` 實心按鈕；在對話頁以 **`button-send-message`（送出）** 為最明確主 CTA。

### 次按鈕（Secondary）
- **定義**：輔助主流程、常見但不結束任務，例如：`outline`／`secondary` 的指派、改派、查訂單、儲存區塊設定、測試連線。
- **位置**：與主 CTA 同區時應在主 CTA 旁、視覺權重較低。

### 危險按鈕（Danger）
- **定義**：刪除品牌／渠道／成員／規則、解除指派、轉人工（中斷 AI 主導）、隱藏留言等不可逆或業務敏感動作。
- **區隔**：實作上常為紅色 icon ghost、或語意明確的 `destructive`／藍色「標記待人工」等需與「一般儲存」分開；**轉人工／恢復 AI** 應成對出現且不可互相遮擋。

### 輔助按鈕（Utility）
- **定義**：複製選單、釘選聯絡人、重新整理大頭照／健康、抽查收合、日期選擇、顯示／隱藏金鑰。
- **原則**：小尺寸、`ghost` 或 icon，不與主 CTA 同色塊競爭。

### Badge / warning / diagnostics 顯示層級
1. **阻斷級**：全域錯誤邊界、SSE／連線中斷橫條（`chat`）、留言 **失敗告警** 紅條、數據戰情室 **系統健康** 異常態。
2. **決策級**：案件 **人工模式／AI 模式／AI 靜音**、VIP、待回、緊急；留言列表左色條（失敗／高風險／待人工／逾時）。
3. **摘要級**：側欄「我的工作／今日戰情」、留言 **戰情摘要** 數字按鈕、inbox 健康統計九宮格。
4. **可下沉級**：規則測試器的詳細命中表、已完成抽查區（可預設收合，但入口要留）。

## B. 各「首頁」首屏原則

### 實際路由首頁：`/`（即時客服）
首屏應優先讓使用者看到：**目前 viewMode／平台篩選語意**、**聯絡人搜尋**、**列表第一屏案件與狀態色塊**；選中案件時 **對話串** 與 **輸入列（送出）** 必須同屏可及；**人工/AI 模式與轉人工／恢復 AI** 不可藏到二層以下。

### 自動客服「監控首頁」：`/comment-center/inbox`
首屏應優先：**失敗告警與錯誤原因**（若有）→ **今日健康統計** → **戰情摘要（例外／失敗／待人工等）** → **列表＋詳情**；**「待補資料」** 標籤不可被視覺弱化到忽略。

### 主管後台「設定入口」：`/settings`
首屏：**系統名稱／Logo** 與 **安全測試模式**（僅 super_admin）為高敏區，其餘區塊往下排即可。

## C. Stitch 最容易誤刪的區塊（高風險）

| 區塊 | 原因 |
|------|------|
| 留言 inbox 健康九宮格＋規則命中／完成類型分布 | 易被當「裝飾 KPI」刪除，實為營運判讀核心 |
| 戰情摘要可點數字篩選 | 不是裝飾，是篩選器 |
| 敏感件「安撫／隱藏／導 LINE／待人工」四格狀態 | 證據鏈，不可改成一句話帶過 |
| 規則頁「五桶順序」說明＋測試器結果（採用原因、候選命中） | AI 美化常併成空泛「分析」 |
| 即時客服：查單三模式按鈕、訂單結果區 | 易被收進選單導致查單路徑變長 |
| `analytics` 的 `section-system-health`、`section-transfer-reasons-health` | 診斷用途，不可刪 |
| `performance` 主管「異常與警示」Card | 與戰情板主卡同級重要 |
| 側欄品牌選擇器與渠道數 | 全域 scope，刪掉會誤以為單一品牌 |
| 粉專與 LINE 設定表（九欄） | 資訊密但不可改成「卡片一句話」後遺失欄位語意 |

## D. 第一輪最值得丟進 Stitch 的頁面（前 5）

1. **`/` 即時客服** — 產品核心三欄、按鈕與風險狀態最多，改版收益最大但誤傷風險最高，需先對齊規格。
2. **`/comment-center/inbox`** — 監控台＋詳情＋AI 建議與人工操作並存，最易被「美化」破壞流程順序。
3. **`/comment-center/rules`（含三子 Tab）** — 規則／mapping／風險三區並列，Stitch 需理解資訊密度與測試證據區。
4. **`/analytics`** — 圖表多但 **`section-system-health` 不能沒**。
5. **`/settings`** — 表單型＋超管測試模式；與 **`/settings/brands-channels`** 可分第二輪，但設定總覽優先建立視覺層級即可。
