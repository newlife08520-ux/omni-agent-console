# 圖文一起理解：現況說明與下一階段規格

## 一、這版（第一階段）到底做到了什麼、沒做到什麼

### 已做到（不要再被圖片模板毀掉）

- **有圖 + 有文字時**：不再被「圖片＋極短／模糊文字」固定模板短路，**一律先進主流程**（state → plan → LLM）。
- 所以體感上不會再出現：客人說「跟你們訂貨要多久」還被回「已收到您的圖片～…①目前是屬於哪一類呢？」。

### 還沒做到（不算真正的看圖說故事）

- **主流程並沒有「吃」最近那張圖的 vision 判讀結果。**
- 具體差距如下。

| 項目 | 現況 | 你要的「圖文一起理解」 |
|------|------|------------------------|
| 近期圖片的處理方式 | 主流程組 `chatMessages` 時，若 `recentMessages` 裡有 `message_type === "image"` 且 `image_url`，會用 `imageFileToDataUri(image_url)` 轉成 data URI；**若成功**，該則會以「文字 + image_url」multimodal 形式送進 LLM。也就是說：**有機會**把「那張圖」當成對話歷史的一則訊息一起送給模型。 | 除了圖本身（若有的話），主流程還要拿到**最近那張圖的 vision 判讀結果**（意圖、訂單資訊、商品資訊、問題類型），並**明確注入** system/context，讓模型「邊看圖的結論、邊接文字」理解。 |
| Vision 結果從哪來 | **純圖片**時（webhook 收圖）：會跑 `handleImageVisionFirst`，得到 `intent`、`confidence`、`reply`，但**只用在當下回覆與 log**，**沒有存成「這則圖片的判讀結果」**供後續主流程使用。**有圖 + 後面補文字**時：主流程不會再去跑 vision，也不會讀取任何先前存下來的 vision 結果。 | 使用者最近貼過圖、接著補文字時，主流程要能拿到：該圖的 vision 判讀結果、圖中抽出的訂單/商品/問題類型、圖片意圖，並進入主流程。 |
| 圖中結構化資訊 | `handleImageVisionFirst` 目前只回傳意圖分類 + 給客人的一句話，**沒有**回傳「從圖中辨識到的訂單編號、手機、商品名稱」等結構化欄位；且該結果也未寫入 DB 或訊息 meta。 | 要能「圖片抽出的訂單資訊／商品資訊／問題類型」進主流程，例如：「我給過了」時能先吃圖裡抽出的欄位，而不是只靠文字。 |

**結論（直接講清楚）**：

- **這版只做到「有文字時不再被固定模板短路」**，沒有做到「主流程真的吃圖的理解結果」。
- 主流程在技術上**有可能**在部分情境下把近期那張圖當 multimodal 送進 LLM（依 `image_url` 是否為本機路徑且 `imageFileToDataUri` 成功），但那只是「順便帶上圖」，**沒有**：
  - 把「最近那張圖的 vision 判讀結果」算出來或從快取讀出，
  - 也沒有把「圖中訂單／商品／問題類型」等結構化資訊注入主流程。
- 因此**不要把這版描述成已經能看圖說故事**；下一階段才要補「圖文一起理解」。

## 二、下一階段目標：圖文一起理解

### 2.1 原則

- 當使用者**最近貼過圖、接著補文字**時，主流程要能拿到並使用：
  - **最近那張圖的 vision 判讀結果**（意圖、信心等），
  - **圖片抽出的訂單資訊／商品資訊／問題類型**（結構化），
  - **圖片意圖**（例如：訂單截圖、客服對話、商品頁、瑕疵圖）。
- 這些**要進入主流程**（system 或專用 context），而不是只有「純圖片」才看圖；這樣才叫真的會看圖、像真人一樣邊看圖邊接文字。

### 2.2 情境與預期行為

| 情境 | 預期 |
|------|------|
| 貼訂單截圖 +「這筆怎麼還沒到」 | 能順著圖裡辨識到的訂單編號／手機接話，直接查單或說明，而不是只把「這筆怎麼還沒到」當一般句子處理。 |
| 貼客服對話截圖 +「你看這個」 | 能理解是在反映前面客服回覆，結合圖的意圖（例如客服對話）與文字接話。 |
| 貼商品頁／尺寸圖 +「這款有現貨嗎」 | 能看懂是在問「這張圖裡這款」的現貨，必要時帶入圖中商品名稱／款式。 |
| 貼圖 +「我給過了」 | 能先吃圖片抽出的欄位（訂單編號、手機、商品等），與 already_provided 流程結合，不要只靠文字搜尋。 |

### 2.3 功能需求（規格要補的項目）

1. **「最近那張圖」的 vision 結果怎麼來**
   - **選項 A**：使用者在 webhook 傳**純圖片**時，除現有 `handleImageVisionFirst` 回覆外，**額外**跑一輪（或擴充現有）vision，產出**結構化結果**（意圖 + 訂單編號、手機、商品名稱、問題類型等），並**存下來**（例如：寫入該則 message 的 meta／或 contact 的「最後一張圖的 vision 結果」）。
   - **選項 B**：不在收圖當下存，而是在**主流程**發現「最近一則 user 是圖、本則 user 是文字」時，**當場**對最近那張圖呼叫 vision（或統一 vision 介面），取得判讀結果 + 結構化抽取，再注入本輪主流程。
   - 不論 A 或 B，主流程在「有近期圖 + 本則文字」時，都必須能拿到：**意圖**、**訂單／手機／商品／問題類型**（若有）。

2. **Vision 輸出要包含的欄位（至少）**
   - **意圖**：例如 `order_screenshot` | `product_issue_defect` | `product_page_size` | `cs_chat` | `off_brand` | `unreadable`（可與現有 handleImageVisionFirst 對齊或擴充）。
   - **結構化抽取**（若意圖適用）：訂單編號、手機號碼、商品名稱／款式、問題類型（瑕疵／尺寸／對話內容摘要等）；可為選填，沒有就空。
   - **confidence**：high / low（可沿用現有）。

3. **進入主流程的方式**
   - 在主流程組 **systemPrompt**（或專用 context）時，若判定「本輪為文字、且近期最後一則 user 為圖片」，則：
     - 取得該圖的 vision 判讀結果（意圖 + 結構化欄位），
     - 注入一段**明確敘述**，例如：
       - 「【客戶上一則傳了圖片】Vision 判讀：意圖=訂單截圖；圖中辨識到：訂單編號 OOO、手機 XXX。請依此與本則文字一起理解並回覆。」
     - 若有訂單／手機／商品，應與現有查單、already_provided、三層搜尋等流程銜接（例如視為「圖裡已提供的資訊」）。

4. **與既有流程的銜接**
   - **already_provided（我給過了）**：若近期圖的 vision 有抽出訂單編號／手機／商品，應納入「已提供資訊」來源（例如三層搜尋的 Layer 2 或專用欄位），不要只靠文字與歷史訊息。
   - **查單**：若圖為 order_screenshot 且已抽出 orderId／phone，主流程或 tool 應能直接使用，減少再問一次。
   - **商品諮詢／現貨**：若圖為 product_page_size 且已抽出商品名稱／款式，主流程應能在回覆中參照「您傳的這款」。

5. **純圖片路徑**
   - 維持現有「純圖片」由 webhook + `handleImageVisionFirst` 處理；可選擇在該路徑**同時**寫入 vision 結果（意圖 + 結構化）供後續主流程使用（即選項 A），以減少「圖 + 文」時重跑 vision 的次數。

## 三、實作時可拆的步驟（建議）

1. **定義「最近一張圖的 vision 結果」的儲存格式與位置**  
   （例如：message meta JSON、或 contact 上 last_image_vision_result。）

2. **擴充 vision 輸出**  
   在現有 `handleImageVisionFirst` 或獨立 vision 呼叫中，增加**結構化抽取**（訂單編號、手機、商品名稱、問題類型），並回傳意圖 + confidence；若採選項 A，在純圖片時寫入儲存。

3. **主流程：偵測「近期圖 + 本則文字」**  
   在 `autoReplyWithAI` 組完 `recentMessages` 後，若「最後一則 user 訊息為圖片」且「本則為文字」，則取得該圖的 vision 結果（從儲存讀取或當場呼叫）。

4. **主流程：注入 vision 結果**  
   將意圖 + 結構化欄位組成一段 system（或 user/context）文字，加入主流程的 systemPrompt；必要時把圖中訂單／手機／商品交給查單、already_provided 等邏輯使用。

5. **銜接 already_provided / 查單 / 商品問答**  
   當 vision 結果帶有訂單／手機／商品時，與 `searchOrderInfoThreeLayers`、tool context、回覆指引接好，讓「我給過了」與「這筆怎麼還沒到」等情境真的用到圖裡資訊。

## 四、小結

- **這版**：只解決「有圖 + 有文字時被固定模板短路」的問題，**沒有**把「最近那張圖的 vision 判讀結果與結構化資訊」注入主流程，因此**不算**真正的看圖說故事。
- **下一階段**：要補「圖文一起理解」—— 主流程在「近期圖 + 補文字」時，必須能拿到並使用該圖的 vision 判讀結果與訂單／商品／問題類型，並進入主流程與查單、already_provided、商品問答銜接，才能像真人一樣邊看圖邊接著文字理解。

此文件可作為下一階段開發的規格依據；實作時可依「儲存格式 → vision 擴充 → 主流程偵測與注入 → 銜接既有流程」的順序拆任務。
