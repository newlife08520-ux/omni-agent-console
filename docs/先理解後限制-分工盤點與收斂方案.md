# 先理解、後限制：分工盤點與收斂方案

## 目標

- **模型先負責**：理解客人現在要什麼、接住上下文、回得自然、好變通。
- **規則只保留**：真正不能踩線的地方做**最後保護**（法務／平台話術／發送前硬擋）。
- **流程原則**：先理解、再決定要不要套規則；不要一開始就被規則蓋掉，導致像真人一樣的理解能力被壓掉。

---

## 一、目前「先限制後理解」的短路分支盤點

以下為目前**在呼叫主流程 LLM 之前**就依規則直接回覆或轉人工、不讓模型參與的分支。依「是否適合保留為先限制」分類。

### 1. 建議保留為「先限制」（真正底線，不交給模型判斷）

| 分支 | 位置 | 觸發條件 | 說明 |
|------|------|----------|------|
| **法務／公關高風險** | routes 約 3643–3679 | `detectHighRisk(userMessage).level === "legal_risk"` | 消保官、檢舉、詐騙、公開、發文、再不處理等。不進 LLM，直接轉人工＋固定告知。**建議保留**：法務/公關紅線不應由模型決定。 |
| **發送前 content guard** | routes 約 4454–4483 | 每則 AI 回覆送出前 | `runPostGenerationGuard`、`runOfficialChannelGuard`、`runGlobalPlatformGuard`：品類禁語、官方渠道禁語、平台話術硬擋。**建議保留**：最後防線，不讓違規話術出去。 |

### 2. 建議改為「先理解後限制」（目前過早短路，壓掉理解）

| 分支 | 位置 | 觸發條件 | 問題 | 收斂方向 |
|------|------|----------|------|----------|
| **安全確認模板（防詐／他平台／待確認來源）** | routes 約 3681–3744 | `classifyMessageForSafeAfterSale(userMessage).matched` 且非連結需求 | 單句關鍵字就回固定模板，不讓模型看上下文（例如先聊訂單再提到「詐騙」一詞可能誤觸）。 | **改為**：仍可做關鍵字偵測，但僅當**高信心**時短路；其餘送進主流程，在 system 註記「若涉及詐騙/他平台，請勿承諾、可引導官方管道」，由模型接住上下文後再決定是否要安全話術；或改為**發送前檢查**：模型回覆若含承諾他平台/防詐敏感句，再替換為安全模板。 |
| **圖片＋極短／模糊文字 → 圖片補充模板** | routes 約 3746–3792 | `hasRecentImageFromUser && isShortOrAmbiguousImageCaption(userMessage)` 且非 human_request、上一則未回過圖片模板 | 有圖＋短句就固定回「已收到您的圖片～…①目前是屬於哪一類呢？…」，不進 LLM。客人若已說「訂貨要多久」仍可能被當短句而重複模板。 | **改為**：**先走主流程**，把近期圖片與本則文字一併當 context 送進 LLM，由模型理解意圖後回覆；僅在**模型回覆後**若判定為「未釐清意圖／需補資訊」再套用「引導分類＋補資訊」的**建議句**或 fallback；或保留為「僅在純圖片、且無任何文字」時才用固定模板，其餘一律先理解。 |
| **尷尬／重複轉人工** | routes 約 3846–3898 | `shouldHandoffDueToAwkwardOrRepeat(...).shouldHandoff` | 規則判斷「同一資料重問兩次、同模板重複、已給過但 AI 答錯、類別跳錯」即直接轉人工，不給模型一次接話機會。 | **改為**：可保留「連續 N 次重複／已給過仍討」作為**升級條件**，但**第一次**先讓模型看到完整上下文（含「我給過了」），由模型嘗試修正（例如道歉＋直接查單）；若模型仍回出不合適內容，再依規則轉人工或發送前替換為轉接句。 |
| **plan.mode === handoff 短路** | routes 約 3901–3956 | `buildReplyPlan` 產出 `mode === "handoff"` | 不進 LLM，直接送一則固定 handoff 句。意圖雖已判為要真人，但**回覆語氣**被固定，無法像真人一樣自然過渡。 | **改為（選做）**：handoff 時仍進 LLM，system 註記「本輪僅做轉接告知，請用一則簡短自然句告知會轉真人，不要多問、不要補查單」；模型產出後再跑 content guard；若逾時或失敗再 fallback 固定句。可保留「不進 LLM」為精簡方案，但若希望更自然，可採此方向。 |
| **return_form_first 固定表單句** | routes 約 3959–3990 | `plan.mode === "return_form_first"` | 商品問題型退換貨直接回固定表單說明，不進 LLM。 | **改為（選做）**：先讓模型依上下文產出承接句（道歉＋表單連結），再在發送前確保必含表單連結；若模型未含則補上或替換。 |
| **off_topic_guard 固定邊界句** | routes 約 3993–4022 | `plan.mode === "off_topic_guard"` | 品牌外問題（晚餐吃什麼、電影推薦等）直接回固定「不陪聊」短句。 | **改為**：仍可判 off_topic，但送進 LLM，system 註記「品牌外生活話題，請簡短友善收邊界、不展開」，由模型產出一句自然收斂；發送前再檢查未偏題即可。 |
| **already_provided 三層搜尋後直接轉人工** | routes 約 4085–4128 | `isAlreadyProvidedMessage(userMessage)` 且三層搜尋無結果 | 客戶說「我給過了」且搜尋不到就直接轉人工，不讓模型嘗試安撫或再確認。 | **改為**：三層仍找不到時，可先讓模型產出一句「抱歉沒找到，幫您轉專人」類自然句，再轉人工；或保留直接轉人工但**文案**由模型生成（僅生成一句）。 |
| **LINE Webhook 轉人工關鍵字** | routes 約 4684–4692 | 文字訊息含 `HUMAN_KEYWORDS` | 不呼叫 `autoReplyWithAI`，直接 handoff。 | **改為**：仍可設 needs_human 並走 handoff，但改為進入統一 `autoReplyWithAI` 流程，由 state/plan 決定 handoff，讓 handoff 文案可與他渠道一致且可選由模型生成（若採 handoff 進 LLM）。 |

### 3. 貼圖／表情符號

| 情境 | 現況 | 建議 |
|------|------|------|
| **LINE 純貼圖** | Webhook 只存 `[貼圖]`／貼圖圖檔，**不觸發** `autoReplyWithAI`，故無回覆。 | 若日後要對貼圖回覆：改為觸發主流程，將貼圖當成「一則多模態訊息」送進 LLM（含圖），由模型依上下文理解情緒／意圖後回一句自然話，**不要**用固定「收到您的圖片」類模板。 |
| **貼圖＋文字** | 文字會觸發 `autoReplyWithAI`，若近期有圖會受「圖片＋短句」分支影響。 | 同上：圖片＋短句分支收斂為「先理解後限制」後，貼圖＋文字自然由模型一起解讀。 |

### 4. 情緒觸發（frustrated / angry）

| 現況 | 說明 |
|------|------|
| **conversation-state-resolver** | `detectEmotion()` 依關鍵字設 `customer_emotion`（frustrated / angry / high_risk）。 |
| **reply-plan-builder** | `needs_human && human_reason === "legal_or_reputation_threat"`（對應 high_risk）→ handoff。frustrated/angry 不直接變 handoff，但會影響 `buildHandoffReply` 的語氣。 |
| **問題** | 若未來有「frustrated 即短路轉人工」的規則，會變成先限制。目前 high_risk 已短路、frustrated 主要影響語氣，相對合理。 |

**建議**：  
- **high_risk**（法務/公關）維持先限制、直接轉人工。  
- **frustrated / angry** 不新增「一律短路轉人工」；維持由 state 傳給 plan，讓模型在 system 裡被要求「先安撫、再處理」，必要時再由 handoff 條件或發送前規則接手。

## 二、意圖／state／plan 的角色（先理解層）

- **resolveConversationState**：用規則從 `userMessage` + 近期對話產出 `primary_intent`、`customer_emotion`、`needs_human` 等，供 **reply-plan** 與後續使用。  
- **buildReplyPlan**：依 state 產出單一 `mode`（handoff / return_form_first / order_lookup / off_topic_guard / answer_directly 等）。

**原則**：  
- 這些可視為「**理解輔助**」：幫模型縮小範圍、鎖定目標，而不是取代理解。  
- 盡量不要用 state/plan **直接取代回覆**（固定句、不進 LLM）；改為把 plan 當成 **system 的約束**（本輪目標、禁止內容、建議語氣），讓模型在約束下**自己生成**自然句。  
- 真正「不能踩線」的（法務、平台話術、品類禁語）留在 **發送前 guard**，不交給模型自由發揮。

## 三、收斂後理想分工（一版方案）

### 1. 先理解（模型負責）

- **輸入**：本則訊息（含貼圖/圖片若可送）＋近期對話＋必要時案件標籤／商品範圍。
- **約束**：system 內註記本輪 **goal**（查單／退貨／轉接／收邊界等）、**禁止內容**（平台話術、品類、推銷等）、**語氣建議**（安撫、簡短、自然過渡）。
- **輸出**：模型產出**一則**自然回覆（可呼叫查單／轉人工等 tool，但回給客人的一句話由模型生成）。

### 2. 再限制（規則只做最後保護）

- **發送前必做**：  
  - 法務/公關高風險：**不送進模型**，直接轉人工＋固定告知（保留現有短路）。  
  - 每則回覆經過：`runPostGenerationGuard`、`runOfficialChannelGuard`、`runGlobalPlatformGuard`；不合規則就清洗或 fallback 短句。
- **可選**：  
  - 若模型回覆被判定為「未釐清意圖／需補資訊」（例如圖片情境下回覆過短或未引導），再套一層「建議補問」或 fallback 模板，而不是**在進模型前**就整則替換成模板。

### 3. 短路保留清單（僅限真正底線）

- 法務／公關 **legal_risk**：直接轉人工，不進 LLM。  
- 發送前 **content guard**：保留，不刪。  
- 其餘（圖片模板、短句模板、off_topic 固定句、return_form_first 固定句、handoff 固定句、尷尬轉人工、already_provided 直接轉人工）：改為「**先進模型理解，再依結果決定是否套規則或替換**」。

## 四、實作優先順序建議

1. **第一階段（高影響、風險可控）**  
   - **圖片＋短句**：改為「僅純圖片且無文字」才用固定模板；其餘（圖片＋任何文字）一律進主流程，把圖與文當 context，由模型回覆；必要時在**回覆後**再判斷是否補一句引導。  
   - **off_topic_guard**：改為進 LLM，system 註記「品牌外話題，簡短友善收邊界」，回覆再 guard。  
   - **安全確認（防詐/他平台）**：提高觸發門檻或改為「高信心才短路」；其餘進主流程並在 system 註記勿承諾、可引導官方，或改為發送前檢查再替換。

2. **第二階段（體驗優化）**  
   - **handoff**：可選讓 handoff 時也進 LLM 產一句自然過渡，再轉人工。  
   - **return_form_first**：可選讓模型產承接句，發送前確保含表單連結。  
   - **尷尬／重複**：第一次先讓模型接「我給過了」等，再決定是否升級轉人工。

3. **第三階段（貼圖／多模態）**  
   - LINE 純貼圖若要回覆：觸發主流程並把貼圖當多模態輸入，由模型依上下文回一句，不套圖片固定模板。

---

## 五、總結一句

- **現在**：很多分支是「先規則、先模板、先短路 → 模型沒機會理解」。  
- **目標**：改成「先讓模型理解與接話，規則只做不能踩線的最後保護與發送前把關」，回話才會像聰明的真人一樣好變通，又有一致的安全邊界。

此文件可作為後續改動的依據；實作時可依階段在 routes、conversation-state-resolver、reply-plan-builder、content-guard 之間逐步收斂，並用既有 phase1/phase2 驗收補情境。
