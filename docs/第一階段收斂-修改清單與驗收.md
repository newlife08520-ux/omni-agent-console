# 第一階段收斂：修改清單、驗收案例與風險評估

## 一、具體修改清單

### 1. 圖片＋短句（僅純圖片才固定模板，有文字一律進主流程）

| 項目 | 檔案 | 修改內容 |
|------|------|----------|
| 移除文字路徑之圖片短路 | `server/routes.ts` | 刪除「圖片＋極短／模糊文字」整段分支（約 50 行）：`hasRecentImageFromUser && isShortOrAmbiguousImageCaption(userMessage)` 時直接回固定模板並 `return` 的邏輯。 |
| 註記 | `server/routes.ts` | 改為註解說明：本流程為「有文字」的訊息，一律先進主流程理解；僅「純圖片、完全無文字」才允許固定模板（該情境在 webhook 圖片分支處理）。 |
| 清理 import | `server/routes.ts` | 自 `safe-after-sale-classifier` 移除未使用之：`isShortOrAmbiguousImageCaption`、`getImageDmReplyForShortCaption`、`getImageDmTemplateNameForShortCaption`、`isImageSupplementContent`。 |

**未改動**：LINE/FB webhook 收到**純圖片**時仍走既有 `handleImageVisionFirst`（或固定 fallback），維持「純圖片無文字」之處理方式。

---

### 2. off_topic（改由模型簡短收邊界，不用固定公告句）

| 項目 | 檔案 | 修改內容 |
|------|------|----------|
| 移除 off_topic 短路 | `server/routes.ts` | 刪除 `plan.mode === "off_topic_guard"` 時直接送 `OFF_TOPIC_GUARD_MESSAGE` 並 `return` 的整段（約 30 行）。 |
| 改走 LLM | `server/routes.ts` | off_topic_guard 時不再 return，改為與其他 mode 一起進入後續 `product_scope_locked` / `goalLocked` / `systemPrompt` / LLM 流程。 |
| product_scope / goalLocked | `server/routes.ts` | 在「Phase 2 product_scope_locked」與 `goalLocked` 的 if-else 中新增 `plan.mode === "off_topic_guard"`：清除 `product_scope_locked`、`customer_goal_locked`，`goalLocked = null`。 |
| system 指引 | `server/routes.ts` | 建 systemPrompt 時若 `plan.mode === "off_topic_guard"`，追加一段：【本輪 品牌外話題】此為品牌服務範圍外的問題…請用一兩句簡短、友善地收邊界…不得推薦菜單、餐廳、電影等…回覆約 30～50 字即可。 |
| 清理 import | `server/routes.ts` | 自 `phase2-output` 移除未使用之 `OFF_TOPIC_GUARD_MESSAGE`。 |

**未改動**：`OFF_TOPIC_GUARD_MESSAGE` 常數仍保留於 `phase2-output.ts`（可供 fallback 或文件參考）；`conversation-state-resolver` 之 `off_topic` 意圖與 `reply-plan-builder` 之 `off_topic_guard` mode 不變。

---

### 3. 安全確認模板（提高短路門檻，避免正常對話誤傷）

| 項目 | 檔案 | 修改內容 |
|------|------|----------|
| 來源不明用語 | `server/safe-after-sale-classifier.ts` | 新增常數 `SOURCE_AMBIGUITY_PHRASES`：哪裡買、哪買、哪個平台、什麼平台、哪個網站、在哪買、這邊買、這裡買、那裏買、別的平台、其他平台買、不是官網、不是你們、哪邊買、什麼通路。 |
| 提高 safe_confirm_order 門檻 | `server/safe-after-sale-classifier.ts` | `classifyMessageForSafeAfterSale` 中，對 `safe_confirm_order` 之觸發改為：除「無本店提示 ＋ 命中 ORDER_SOURCE_AMBIGUOUS_KEYWORDS」外，**尚須**訊息中至少出現一個 `SOURCE_AMBIGUITY_PHRASES` 才回傳 matched。 |

**未改動**：詐騙冒用（`fraud_impersonation`）、他平台（`external_platform`）之觸發條件與優先順序不變；僅 `safe_confirm_order` 提高門檻。

## 二、驗收案例

### A. 圖片＋短句收斂

| 情境 | 步驟 | 預期結果 |
|------|------|----------|
| 圖＋「你好」 | 用戶先傳一張圖，再傳文字「你好」。 | 不再回固定「已收到您的圖片～…①目前是屬於哪一類呢？…」；改走主流程，由 LLM 依上下文回覆（可為招呼或引導）。 |
| 圖＋「跟你們訂貨要多久」 | 用戶先傳圖，再傳「跟你們訂貨要多久」。 | 不再回圖片補充模板；改走主流程，回覆出貨／訂貨時間相關內容。 |
| 純圖片（無文字） | 用戶只傳一張圖、不傳文字。 | 仍由 webhook 圖片分支處理（vision-first 或 fallback），行為與改動前一致。 |

### B. off_topic 收斂

| 情境 | 步驟 | 預期結果 |
|------|------|----------|
| 晚餐吃什麼好 | 用戶傳「晚餐吃什麼好」。 | plan.mode 仍為 `off_topic_guard`；**不回**固定句「這部分比較不是我們服務範圍～…」；改由 LLM 產出一句簡短、友善收邊界（約 30～50 字），且**不推薦**菜單／餐廳／電影。 |
| 電影推薦 | 用戶傳「有電影推薦嗎」。 | 同上，回覆為簡短收邊界、不展開推薦。 |

### C. 安全確認門檻

| 情境 | 步驟 | 預期結果 |
|------|------|----------|
| 我的訂單還沒到 | 用戶傳「我的訂單還沒到」。 | **不**觸發 safe_confirm 模板；走一般查單／售後流程。 |
| 訂單是哪裡買的可以退嗎 | 用戶傳「訂單是哪裡買的可以退嗎」。 | 觸發 safe_confirm_order，回安全確認模板（引導釐清來源／管道）。 |
| 有私人帳號叫我匯款 | 用戶傳「有私人帳號叫我匯款，這是你們的嗎」。 | 仍觸發 fraud_impersonation，行為與改動前一致。 |
| 在蝦皮買的 | 用戶傳「在蝦皮買的訂單有問題」。 | 仍觸發 external_platform，行為與改動前一致。 |

### D. 未改動部分（迴歸）

| 項目 | 預期 |
|------|------|
| 法務高風險 | `legal_risk` 仍不進 LLM，直接轉人工＋固定告知。 |
| 發送前 guard | `runPostGenerationGuard`、`runOfficialChannelGuard`、`runGlobalPlatformGuard` 仍於每則回覆送出前執行。 |
| handoff | plan.mode === handoff 時仍不進 LLM，維持固定 handoff 句＋轉接。 |

## 三、風險評估

| 風險 | 等級 | 說明 | 緩解 |
|------|------|------|------|
| 圖片＋文字改走 LLM 後回覆不穩 | 中 | 有圖＋短句不再套固定模板，改由模型產出，可能偶爾過長或偏題。 | 沿用既有發送前 guard 與長度/禁語限制；可觀察一至兩週，必要時在 system 中加強「有圖時簡短承接、必要時再引導分類」等指引。 |
| off_topic 由模型收邊界偶爾推薦到餐廳/電影 | 低 | 若模型未完全遵守「不得推薦」可能出現一次半次。 | system 已明確寫「不得推薦菜單、餐廳、電影等」；發送前 guard 可視需要新增「品牌外推薦」類規則攔截。 |
| safe_confirm 門檻提高後漏攔 | 低 | 極少數「只講訂單/退款但沒講哪裡買」卻在問他平台的情境，可能不再觸發 safe_confirm。 | 此類情境改由主流程 LLM 處理，可在 system 中保留「若涉及他平台或來源不明，勿承諾、可引導官方管道」；詐騙與他平台關鍵字仍先短路，風險可控。 |
| 依賴 LLM 延遲與可用性 | 中 | off_topic、圖＋文皆多走一次 LLM，延遲略增、且依賴 API 可用。 | 與現有主流程一致，無額外端點；若 API 逾時既有流程已有轉人工／重試。 |

整體：**法務高風險與發送前 guard 未動**，安全底線不變；體感與真人感提升，風險以中低為主，可依上表觀察與補強。

## 四、小結

- **第一優先**：圖片＋短句 → 僅純圖片才固定模板，有文字一律進主流程。✅  
- **第二優先**：off_topic → 改由模型簡短收邊界，不再固定公告句。✅  
- **第三優先**：安全確認 → 提高 safe_confirm_order 短路門檻（須含來源不明意圖）。✅  
- **未改動**：法務高風險、發送前 guard、handoff 仍維持現狀。

以上為「只做第一階段」之具體修改清單、驗收案例與風險評估。
