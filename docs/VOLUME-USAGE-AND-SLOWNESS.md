# Volume 使用率 94% 原因與載入變慢說明

## 一、Volume 裡主要吃空間的是什麼？

`omni-agent-console-volume`（對應 `DATA_DIR`，production 預設 `/data`）底下會長大的只有這幾類：

| 來源 | 路徑/檔案 | 說明 |
|------|-----------|------|
| **SQLite 資料庫** | `omnichannel.db` | 聯絡人、訊息、AI 日誌、Meta 留言、案件、分配紀錄等全部存在同一個 DB。訊息與日誌隨時間只增不減，DB 會持續變大。 |
| **上傳與下載檔案** | `uploads/` | 使用者上傳的圖片/檔案、LINE/FB 下載的圖片與影片（`line-*.jpg`、`fb-*.jpg`、`.mp4` 等）、素材庫圖片、頭像等，全部寫在 `uploads/`。 |
| **WAL / 暫存** | `omnichannel.db-wal`、`omnichannel.db-shm` | SQLite 的 WAL 模式會多出這兩個檔案，隨寫入量變大。 |

沒有把 log 寫到這個 volume 的設計，所以 **94% 主要是 DB + uploads**。

---

## 二、為什麼會「載入訊息好慢、卡頓」？

Volume 用到 **94%（約 450–470 MB / 500 MB）** 時，容易造成：

1. **磁碟 I/O 變慢**  
   空間所剩無幾時，寫入與讀取都會變慢，訊息列表、AI 回覆、訂單查詢都會等 DB 與檔案 I/O。

2. **SQLite 效能**  
   - 單一 `omnichannel.db` 若破百 MB、又沒有定期整理，查詢與寫入都會變慢。  
   - 訊息一多，`getMessages`、`getContacts`（含最後一則訊息）等都會受影響，體感就是「載入訊息好慢、會卡頓」。

3. **沒有多餘空間**  
   系統與應用暫存、WAL 成長空間不足，可能觸發更多 I/O 或錯誤，進一步拖慢整體。

所以 **「volume 快滿」和「載入訊息慢、卡頓」是高度相關的**，你的體感是合理的。

---

## 三、建議做法（先降用量再談優化）

### 1. 立即：加大 Volume（例如升級到 +5 GB）

- 在 Railway（或你用的平台）把 `omni-agent-console-volume` **升級到 5 GB**，先讓使用率從 94% 降到約 10–20%。  
- 這樣可以立刻減輕 I/O 壓力，載入訊息與操作通常會明顯變順。

### 2. 中期：減少持續成長

- **訊息 / AI 日誌**  
  - 若法規或營運允許，可考慮：  
    - 只保留最近 N 天/ N 萬則訊息，其餘歸檔或刪除；或  
    - 對 `ai_logs`、舊 `messages` 做定期清理或歸檔。  
  - 需在應用或排程裡實作對應的 DELETE / 歸檔邏輯，並注意備份與法遵。

- **uploads**  
  - 檢查 `uploads/` 是否有過多、過大的 LINE/FB 下載檔（圖片、影片）。  
  - 可訂策略：例如只保留最近 90 天、或超過一定數量/容量就刪除不再使用的檔案（需與 DB 中 `message.image_url` 等欄位對應，避免破圖）。

- **SQLite 整理**  
  - 在低峰時偶爾執行 `VACUUM`（或 `PRAGMA optimize`），可回收空間、改善查詢效能。  
  - 需注意 VACUUM 會暫時鎖庫，要排程在流量低時做。

### 3. 長期：監控與告警

- 對 volume 使用率設告警（例如 > 80% 就通知），避免再次接近滿盤。  
- 若之後有「只查近期訊息」的需求，可搭配既有分頁（before_id / limit）與前端 infinite scroll，減少單次查詢量，對 DB 與載入體感都有幫助。

---

## 四、總結

- **94% 高的原因**：主要是 **`omnichannel.db`** 與 **`uploads/`**（含 LINE/FB 下載的圖片/影片）隨時間只增不減。  
- **載入訊息慢、卡頓**：在 volume 快滿的情況下，磁碟 I/O 與 SQLite 效能都會變差，和目前現象一致。  
- **建議**：先 **升級 Volume（例如 +5 GB）** 把使用率降下來，再規劃訊息/日誌與 uploads 的清理或歸檔策略，並視需要做 SQLite 的 VACUUM 與監控告警。
