# R1 遮罩真實情境對照（≥5 則）

> 訂單號、手機、姓名皆已遮罩；重點在 **政策／話術** 修前修後。

## 1. 官網意圖 + Shopline 未設定

| | 內容 |
|---|------|
| **修前** | 客戶：「我要查官網訂單 09********」→ 可能出現「這支手機（官網）查無」，**像有查 API**。 |
| **修後** | 明確說明 **官網 API 未設定，無法即時代查**，並引導單號／商品+手機／全部訂單語意。 |

## 2. 官網 + 商品手機，Shopline 無結果

| | 內容 |
|---|------|
| **修前** | 可能 **回落一頁**，把非官網單當成回覆。 |
| **修後** | **僅 Shopline 路徑**；無結果即查無，**不冒充官網**。 |

## 3. 純手機（未說「全部訂單」）

| | 內容 |
|---|------|
| **修前** | 模型或 fast path 可能 **直接單筆定案**。 |
| **修後** | 要求 **商品+手機** 或「查我全部訂單」等語意；**不單筆定案**。 |

## 4. `local_only` 單筆索引命中

| | 內容 |
|---|------|
| **修前** | 「幫您查到了」+ 完整卡，**過度確定**。 |
| **修後** | **候選摘要** + 需 live 確認／補問是否還有其他訂單。 |

## 5. 久候／何時出貨（待出貨狀態）

| | 內容 |
|---|------|
| **修前** | 「目前狀態：待出貨」等 **後台口吻** 硬唸。 |
| **修後** | 統一走 **品牌久候模板**（道歉、5 工作天／7–20 工作天、不加壓確切日、可催急）；**不**在模板路徑唸「目前狀態：待出貨」。 |

## 6. COD 被說成付款失敗（加列）

| | 內容 |
|---|------|
| **修前** | 一頁 `pending`+超商 被誤判 failed。 |
| **修後** | `derivePaymentStatus` **先 COD**；對客 **貨到付款（到收／取件時付款）**。 |

## 7. LINE Pay 失敗被說成待付款（加列）

| | 內容 |
|---|------|
| **修前** | raw 已失敗仍 pending。 |
| **修後** | `payment_status_raw` 失敗訊號 → **failed**，對客「付款失敗／訂單未成立」。 |
