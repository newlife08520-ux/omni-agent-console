# 售後與出貨邏輯 — 強制驗收清單

本輪**不改功能**，僅依現有 reply-plan-builder / routes / handoffBlock 規則做正式驗收。  
驗收時：用「使用者輸入」實際送進系統，對照「預期 plan.mode / 查單 / 表單 / 轉人工」與「預期回覆特徵」逐項打勾。

---

## 特別驗收規則（必過）

| 規則 | 驗收方式 |
|------|----------|
| 久候型不能一開始就丟表單 | 久候／想取消類輸入，第一則 AI 回覆不得出現表單連結或「請填寫退換貨表單」為第一句。 |
| 商品問題型一定要正式售後 | 瑕疵／損壞／錯貨／缺件類輸入，AI 要先道歉並導向表單或正式售後，不得只說「會盡快出貨」「7–20 工作天」當主軸。 |
| 未查單前不能太早提其他平台 | 在尚未用訂單查詢工具且查無此筆前，回覆中不得出現「其他平台」「若非官方通路」「建議向該平台客服確認」。 |
| 久候型要優先安撫與留單 | 久候／想取消類，回覆要有安撫句＋查詢或說明現貨／加急／7–20 天，語氣像在留單而非推表單。 |
| 語氣不能太像制度公告 | 不得整段像公告（如「若您是透過官方通路…」「請依照規範填寫」「系統查無資料」），要像真人客服。 |

## 情境一：久候但還沒爆炸（先安撫、先查、不立刻表單／轉人工）

### 1-1

| 項目 | 內容 |
|------|------|
| **使用者輸入** | 怎麼還沒出貨 |
| **預期 plan.mode** | `order_lookup` 或 `aftersales_comfort_first`（若被判為售後意圖則為後者） |
| **預期是否查單** | 是（可主動詢問訂單編號／商品＋手機並查詢） |
| **預期是否給表單** | 否 |
| **預期是否轉人工** | 否 |
| **預期回覆必須出現的重點** | 先一句承接或安撫（如：不好意思讓您久等了／了解，這邊先幫您確認）；說明會查詢／盡快安排或若缺貨約 7–20 工作天；可出現「會盡快」「備註加急」等。 |
| **預期回覆禁止出現的內容** | 第一句就給表單連結；「請填寫退換貨表單」當開頭；「若您是透過官方通路／其他平台」；直接轉人工的用語。 |

### 1-2

| 項目 | 內容 |
|------|------|
| **使用者輸入** | 我等好幾天了 |
| **預期 plan.mode** | `order_lookup` 或 `aftersales_comfort_first` |
| **預期是否查單** | 是 |
| **預期是否給表單** | 否 |
| **預期是否轉人工** | 否 |
| **預期回覆必須出現的重點** | 安撫（如：不好意思讓您久等了）；幫您確認出貨狀況／查詢；若有現貨會盡快／若為預購約 7–20 工作天；會盡量協助加快。 |
| **預期回覆禁止出現的內容** | 一開頭就表單；其他平台／官方通路；立刻轉人工。 |

### 1-3

| 項目 | 內容 |
|------|------|
| **使用者輸入** | 不是現貨嗎 |
| **預期 plan.mode** | `order_lookup` 或 `aftersales_comfort_first` |
| **預期是否查單** | 是（說明現貨／預購差異並可查詢） |
| **預期是否給表單** | 否 |
| **預期是否轉人工** | 否 |
| **預期回覆必須出現的重點** | 承接／安撫；說明現貨會盡快或目前排單狀況；可查詢該筆訂單。 |
| **預期回覆禁止出現的內容** | 第一句表單；其他平台；制度公告式說法。 |

## 情境二：客人說不想等了（先承接、先留單，不秒丟表單）

### 2-1

| 項目 | 內容 |
|------|------|
| **使用者輸入** | 我不要了 |
| **預期 plan.mode** | `aftersales_comfort_first`（原因未明或久候型） |
| **預期是否查單** | 可查（先理解原因、可查詢出貨能否加急） |
| **預期是否給表單** | 否（第一輪不給） |
| **預期是否轉人工** | 否 |
| **預期回覆必須出現的重點** | 先承接／理解（如：了解、不好意思）；可詢問是否因等太久或想取消；可主動查詢出貨、說明能否加急或約 7–20 天，嘗試留單。 |
| **預期回覆禁止出現的內容** | 一開口就「請填寫退換貨表單」或表單連結；「若為其他平台購買」；立刻轉人工。 |

### 2-2

| 項目 | 內容 |
|------|------|
| **使用者輸入** | 太久了我想取消 |
| **預期 plan.mode** | `aftersales_comfort_first` |
| **預期是否查單** | 是 |
| **預期是否給表單** | 否（第一輪） |
| **預期是否轉人工** | 否 |
| **預期回覆必須出現的重點** | 安撫（不好意思讓您久等）；先幫您確認出貨狀況／能否加急；說明現貨會盡快或預購約 7–20 工作天；會盡量協助加快。 |
| **預期回覆禁止出現的內容** | 第一句就丟表單；先講其他平台；冷硬的「請填寫取消表單」。 |

### 2-3

| 項目 | 內容 |
|------|------|
| **使用者輸入** | 你們這樣我要退貨 |
| **預期 plan.mode** | `aftersales_comfort_first`（久候型，非明確堅持） |
| **預期是否查單** | 是 |
| **預期是否給表單** | 否（第一輪先安撫與查詢） |
| **預期是否轉人工** | 否 |
| **預期回覆必須出現的重點** | 先承接與安撫；理解不滿（出貨慢等）；主動查詢、說明能否加急或時程；嘗試留單。 |
| **預期回覆禁止出現的內容** | 因出現「退貨」就立刻導表單；其他平台；一開口就轉人工。 |

## 情境三：商品有問題（正式售後、表單）

| **使用者輸入** | 收到是壞的 |

## 情境四：客人很兇、很堅持（表單或人工）

### 4-1

| 項目 | 內容 |
|------|------|
| **使用者輸入** | 我就是要退 |
| **預期 plan.mode** | `return_form_first` 或 `handoff`（若 needs_human） |
| **預期是否查單** | 否（本輪以表單／人工為主） |
| **預期是否給表單** | 是（可先一句安撫後給表單） |
| **預期是否轉人工** | 可（明確堅持可轉） |
| **預期回覆必須出現的重點** | 簡短承接後，提供表單或安排人工；不反覆嘗試留單拖延。 |
| **預期回覆禁止出現的內容** | 一直只說「先幫您查出貨」「再等 7–20 天」而不給表單或轉人工；冷硬拒絕。 |

### 4-2

| 項目 | 內容 |
|------|------|
| **使用者輸入** | 不要跟我講這些 |
| **預期 plan.mode** | 可能 `handoff`（repeat_unresolved）或 `return_form_first` |
| **預期是否查單** | 否 |
| **預期是否給表單** | 可 |
| **預期是否轉人工** | 可 |
| **預期回覆必須出現的重點** | 不再長篇安撫或查單說明；簡短承接後給表單或轉人工。 |
| **預期回覆禁止出現的內容** | 繼續長篇「會盡快安排」「7–20 工作天」而不提供表單或真人。 |

### 4-3

| 項目 | 內容 |
|------|------|

| **預期 plan.mode** | `return_form_first` 或 `handoff` |
| **預期是否查單** | 否 |
| **預期是否給表單** | 是 |
| **預期是否轉人工** | 可 |

## 情境五：查不到訂單（不早提其他平台）

### 5-1

| 項目 | 內容 |
|------|------|
| **使用者輸入** | 訂單編號 OOO 幫我查（假設編號錯誤或查無） |
| **預期 plan.mode** | `order_lookup` |
| **預期是否查單** | 是（先查） |
| **預期是否給表單** | 否 |
| **預期是否轉人工** | 否（第一次查無時不轉） |
| **預期回覆必須出現的重點** | 若查無：先說「目前這邊暫時還沒查到這筆資料」；再請補資料（訂單編號／商品名稱＋手機等）。 |
| **預期回覆禁止出現的內容** | 第一次查無就說「可能不是我們這邊的訂單」「建議找其他平台客服」「若非官方通路」；未再收資料就轉人工。 |

### 5-2

| 項目 | 內容 |
|------|------|
| **使用者輸入** | 我要查訂單（未提供編號或只給不完整資訊） |
| **預期 plan.mode** | `order_lookup` |
| **預期是否查單** | 先收資料再查（或引導提供訂單編號／商品＋手機） |
| **預期是否給表單** | 否 |
| **預期是否轉人工** | 否 |
| **預期回覆必須出現的重點** | 先承接（如：了解，這邊幫您確認）；禮貌請提供訂單編號或商品名稱＋手機，再查詢。 |
| **預期回覆禁止出現的內容** | 在尚未取得足夠資訊並查詢前就提「其他平台」「官方通路」。 |

### 5-3

| 項目 | 內容 |
|------|------|
| **情境** | 已用訂單編號／商品+手機查過兩次，仍查無，且客人明顯不耐煩 |
| **預期 plan.mode** | 可轉 `handoff` 或維持 order_lookup 但語氣轉為可轉人工 |
| **預期是否查單** | 已查過 |
| **預期是否給表單** | 否 |
| **預期是否轉人工** | 可 |
| **預期回覆必須出現的重點** | 此時才可自然提到「有可能不是從我們這個系統下單」；可建議「若您願意我也可以幫您轉專人再確認」；語氣自然，不系統宣告。 |
| **預期回覆禁止出現的內容** | 在僅查一次或未收齊資料時就說「可能不是我們系統」；冷硬的「系統查無資料，請找該平台」。 |

## 驗收使用方式

1. **對照 plan.mode**：在開發環境可於日誌或除錯輸出中確認 `buildReplyPlan` 產出的 `mode`，與上表「預期 plan.mode」一致。  
2. **對照行為**：實際送訊息後，確認是否有呼叫訂單查詢工具、是否回覆內含表單連結、是否觸發轉人工。  
3. **對照文案**：用「預期回覆必須出現的重點」與「預期回覆禁止出現的內容」逐項檢查 AI 實際回覆。  
4. **特別規則**：最上方五條「特別驗收規則」每條都要過，才算通過本輪售後與出貨邏輯驗收。
