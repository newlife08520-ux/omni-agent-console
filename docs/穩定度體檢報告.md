# 系統穩定度體檢報告（最終稽核）

本報告針對「伺服器 OOM」、「資料庫併發鎖死」、「API / 前端無限迴圈」等致命風險進行掃描結論。

---

## 1. 伺服器 OOM 崩潰風險

| 項目 | 狀態 | 說明 |
|------|------|------|
| GET /api/contacts | ✅ 已防護 | 有 `limit` 參數，預設 100、上限 500，不會一次載入全表。 |
| GET /api/contacts/:id/messages | ✅ 已防護 | 有 `limit`（預設 80、上限 500）、`before_id` 分頁，不會一次拉全量訊息。 |
| storage.getContacts | ✅ 已防護 | 支援 `limit`，且只取聯絡人 + 每聯絡人「最新一則訊息」subquery，不載入全量歷史。 |
| Webhook 處理 | ✅ 已防護 | 事件逐筆處理，無一次性載入整表；LINE/FB 皆為單一 request body 處理。 |
| SSE 廣播 | ✅ 已防護 | `broadcastSSE` 僅遍歷既有 client Set 並 write，不建立大物件或無界陣列。 |

**結論**：未發現會導致單次請求或單一操作無界成長記憶體的路徑，OOM 風險已控制在可接受範圍。

---

## 2. 資料庫併發 / 鎖死 (DB Locked) 風險

| 項目 | 狀態 | 說明 |
|------|------|------|
| 連線 / 寫入模式 | ✅ 單一寫入 | better-sqlite3 同步 API，單 process 單連線，無多 thread 競爭。 |
| GET 是否寫入 | ✅ 已排除 | GET /api/contacts/:id 已移除 `setImmediate` 內之 DB 寫入與廣播，僅讀取。 |
| 長時間交易 | ✅ 未見 | 未在單一 request 內開啟長時間 transaction 不 commit；Webhook 處理為短交易。 |
| busy_timeout | ⚠️ 建議 | 若未來改為多 process / 讀寫分離，建議設定 SQLite `busy_timeout` 以降低鎖衝突。 |

**結論**：目前架構下無明顯「讀取時寫入導致鎖死」或長時間佔鎖路徑；GET 冪等化後不再因「點擊聯絡人」觸發背景寫入。

---

## 3. API / 前端無限迴圈 (Infinite Loop) 風險

| 項目 | 狀態 | 說明 |
|------|------|------|
| GET 觸發寫入 + 廣播 | ✅ 已切斷 | GET /api/contacts/:id 不再執行 `updateContactAiSuggestions` 與 `broadcastSSE("contacts_updated")`，已切斷「讀取 → 寫入 → 廣播 → 重新讀取」死循環。 |
| 前端 SSE useEffect | ✅ 已防護 | 依賴陣列為 `[]`，僅 mount 時綁定；內部僅 `invalidateQueries`，無 `refetchQueries`；cleanup 有 `es?.close()` 與 `clearTimeout`。 |
| 前端 SSE 依賴 | ✅ 已防護 | 使用 `queryClientRef.current`，未將 `selectedId` 等變動值放入 deps，避免點擊聯絡人重綁 EventSource。 |
| 其他 useEffect | ✅ 合理 | 其餘 useEffect 多為單次或依穩定 deps（如 `[queryClient]`、`[]`），未見「setState → re-render → effect 再觸發請求」之迴圈。 |

**結論**：讀寫廣播死循環已排除；前端 SSE 不會因切換聯絡人而重複註冊或觸發大量 refetch，無限迴圈風險已收斂。

---

## 4. 綜合結論

- **OOM**：列表與訊息皆有 limit / 分頁，無單次無界載入；SSE 與 Webhook 未持有無界資料結構。
- **DB Locked**：GET 不再寫入；單 process 單連線，未見長時間佔鎖或明顯鎖競爭。
- **無限迴圈**：GET 冪等、SSE 僅 invalidate 且 deps=[]，已切斷「讀取 → 寫入 → 廣播 → 重新讀取」與「點擊聯絡人 → 重綁 SSE → 暴量請求」兩類死循環。

**目前系統已排除會導致「伺服器 OOM 崩潰」、「資料庫併發鎖死」、「API / 前端無限迴圈」的已知致命架構缺陷。** 後續若新增「背景 AI 建議」等寫入，應僅在 Webhook 新訊息或客服手動觸發時執行，切勿在 GET 或單純讀取流程中寫入 DB 並廣播 SSE。
