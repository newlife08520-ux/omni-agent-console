# 為什麼「幾乎打不開」— 根因說明

## 從你終端 log 看到的狀況

1. **一頁商店銷售頁同步**：每次會打 **133 頁** 一頁商店 API，共 **3974 筆** 銷售頁，且會**重複執行**（開機一次、每小時一次、或有人開「訂單查詢 → 選擇商品」時若快取過期也會觸發）。
2. **GET /api/health/status 200 in 1878ms**：健康檢查有時要 **約 1.8 秒** 才回，代表當時伺服器負載高。
3. **銷售頁快取 同步失敗: api_error_503**：一頁商店 API 有時 503，可能觸發重試，進一步加重負載。

## 根因（簡短）

- **伺服器一啟動**就立刻執行 `refreshPagesCache()`，等於一開機就對外打 **133 次** HTTP 請求。Node 單執行緒，這段期間處理其他請求（登入、聯絡人、首頁）會變慢，體感就是「打不開」或「卡很久」。
- **使用者點「訂單查詢」→「選擇商品（銷售頁）」**時，前端會打 `GET /api/orders/pages`。若此時快取是空的或已過期（1 小時），後端會**同步**執行 133 次請求再回傳，這一筆 HTTP 可能要 **1～2 分鐘以上** 才回來，瀏覽器會一直等，畫面就像當住、打不開。

所以：**不是「整個軟體壞掉」**，而是  
1）開機時被 133 次同步拖慢，  
2）第一次或過期後要載入銷售頁時，單一請求被 133 次同步卡住。

## 軟體其餘部分

- 登入、聯絡人、訊息、SSE、設定等 API 在 log 裡多是 200、幾 ms～十幾 ms，**在沒被銷售頁同步拖到的時候是正常的**。
- 因此「軟體都正常運作嗎」：**在沒有正在跑 133 次銷售頁同步時，其餘功能是正常可用的**；一旦同步在跑或卡住，就會覺得整機卡、幾乎打不開。

## 建議修正（已實作於下一段）

1. **延後開機同步**：不在 `registerRoutes` 一進來就執行 `refreshPagesCache`，改為延遲約 30 秒再在背景執行，讓伺服器先能快速回應登入與首頁。
2. **首筆銷售頁不卡住**：呼叫 `ensurePagesCacheLoaded` 時，若快取為空，**先立刻回傳空陣列**，再在背景觸發 `refreshPagesCache`，下次請求或前端重試就能拿到完整列表；這樣「選擇商品」不會等 1～2 分鐘才開。

上述兩點都不改 UI、不改一頁商店串接邏輯，只調整「何時、如何」載入快取，避免一次 133 次請求拖垮整機與單一請求。
