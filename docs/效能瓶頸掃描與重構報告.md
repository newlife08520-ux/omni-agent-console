# 客服系統效能瓶頸掃描與重構報告

## 掃描範圍與結論摘要

針對「聯絡人/對話列表卡頓、逾 2000 件案件時 UI 明顯延遲」進行四項效能死角掃描，並已完成優先重構。

| 項目 | 發現問題 | 已實作修復 |
|------|----------|------------|
| 1. 列表 API 分頁與 Payload | 有 limit（預設 100、前端傳 300），但**無 offset**；戰情數字 API 曾**全表載入** | ✅ 新增 offset 分頁；戰情改為純 COUNT 查詢 |
| 2. 前端長列表渲染 | **已使用** `@tanstack/react-virtual` 虛擬列表 | 無需改動 |
| 3. 背景輪詢與更新頻率 | 聯絡人列表 **5 秒** 輪詢且 **背景也輪詢**，易造成大量 re-render | ✅ 改為 20 秒、背景不輪詢 |
| 4. DB N+1 與索引 | 戰情數字依賴 `getContacts()` 全表 + 迴圈；**無 N+1**（last_message 批次查）；索引已存在 | ✅ 戰情改為 COUNT，索引沿用 |

---

## 1. 聯絡人/對話列表 API 分頁與 Payload（Backend）

### 掃描結果

- **GET /api/contacts**（`server/routes.ts` 約 L2610）
  - 已有 **limit**：`req.query.limit`，預設 100、上限 500；前端目前傳 `limit=300`。
  - **原本沒有 offset**：無法做「下一頁」分頁，只能一頁 300 筆。
- **Payload 過大風險**
  - 列表 API 本身有 limit，**不會**一次把 2000+ 筆全部回傳。
  - **真正問題**在「戰情數字」API（見下方第 4 點）：`/api/manager-stats`、`/api/agent-stats/me` 曾呼叫 `storage.getContacts(brandId)` 或 `getContacts(undefined, userId, userId)` **未傳 limit**，導致 **全表載入**（2000+ 筆）後在記憶體裡迴圈算數字。

### 已實作修改

1. **GET /api/contacts 支援 offset 分頁**
   - Query 參數：`limit`（1–500）、`offset`（≥0）。
   - `storage.getContacts(..., limit, offset)` 新增第五參數 `offset`，SQL 使用 `LIMIT n OFFSET m`。
   - 前端可維持目前 `limit=300` 不帶 offset（第一頁），之後若要「載入更多」可傳 `offset=300`、`offset=600` 等。

2. **戰情數字改為純 COUNT，不再載入全表**
   - 新增：
     - `storage.getManagerStatsCounts(brandId?)`：傳回 `today_new`、`unassigned`、`closed_today`、`overdue`、`urgent_simple`、`vip_unhandled`，全部以單一 SQL COUNT（及必要子查詢）計算。
     - `storage.getAgentStatsCounts(agentId)`：傳回 `pending_reply`、`closed_today`、`overdue`、`tracking`、`urgent_simple`。
     - `storage.getAgentPendingReplyCount(agentId)`：單一客服「待回覆」數，供 manager-stats 的 `team[].pending_reply` 使用。
   - **/api/manager-stats**：改為只呼叫 `getManagerStatsCounts(brandId)` + 對每位客服呼叫 `getOpenCasesCountForAgent`、`getAgentPendingReplyCount`，**不再**呼叫 `getContacts(brandId)` 或 `getContacts(brandId, m.id)`。
   - **/api/agent-stats/me**：改為只呼叫 `getOpenCasesCountForAgent(userId)` 與 `getAgentStatsCounts(userId)`，**不再**呼叫 `getContacts(undefined, userId, userId)`。

效果：案件數 2000+ 時，戰情 API 不再載入全表，回應時間與記憶體使用可明顯下降。

## 2. 前端長列表渲染效能（Frontend）

### 掃描結果

- 左側聯絡人列表（`client/src/pages/chat.tsx`）**已使用虛擬列表**：
  - `@tanstack/react-virtual` 的 `useVirtualizer`，依 `contactListSafe.length` 做虛擬滾動。
  - `rowVirtualizer.getVirtualItems()` 只渲染可見區塊 + overscan 8，大量聯絡人時不會一次掛載全部 DOM。
- 結論：**無需再引入 react-window**；現有虛擬列表實作正確。

### 可選優化（未強制實作）

- `filteredContacts` 在每次父元件 re-render 時會重新 filter/sort；若 300 筆且 re-render 頻繁，可考慮 `useMemo` 依 `contacts`、`searchQuery`、`viewMode`、`platformFilter` 做依賴，減少重算與虛擬列表的 churn。
- 若未來列表單頁筆數再放大，可再評估將「第一頁」改為較小（例如 100），搭配「載入更多」用 offset 分頁，進一步縮小單次 Payload。

## 3. 背景輪詢與狀態更新頻率（Frontend）

### 掃描結果

- **聯絡人列表**（chat 頁 `useQuery` `/api/contacts`）：
  - `refetchInterval: 5000`（每 **5 秒** 打一次）。
  - `refetchIntervalInBackground: true`（分頁在背景時仍輪詢）。
- **訊息列表**：`refetchInterval: 10000`（10 秒）。
- **主管/客服戰情**（app-sidebar、App.tsx）：`refetchInterval: 15000`（15 秒）。

影響：每 5 秒一次聯絡人 refetch → 整頁 re-render → 左側列表重新 filter/sort → 虛擬列表更新；若同時有 SSE 或其它 invalidate，容易疊加為明顯卡頓。

### 已實作修改

- 聯絡人列表同一個 `useQuery`：
  - `refetchInterval`: **5000 → 20000**（20 秒）。
  - `refetchIntervalInBackground`: **true → false**（分頁在背景時不輪詢聯絡人列表）。
- 其餘（訊息、戰情）維持不變；若仍覺得卡頓，可再將戰情改為 20–30 秒或僅依 SSE 更新。

## 4. 資料庫查詢的 N+1 與索引（DB/ORM）

### 掃描結果

- **N+1**
  - **聯絡人列表** `storage.getContacts()`：**無 N+1**。先查 contacts 一筆 SQL，再依 `contactIds` 一次查「每個 contact 的最後一則訊息」一筆 SQL，再依需要一次查 `getAgentContactFlags`。
  - **戰情**：原本是「一次全表 getContacts + 迴圈」與「每個客服再 getContacts(brandId, m.id)」，屬於**全表/大結果集**問題，不是經典 N+1，但負擔更大；已改為 COUNT API。
- **索引**（`server/db.ts` 的 `ensurePerformanceIndexes()`）
  - 已有：`idx_contacts_brand_id`、`idx_contacts_assigned_agent_id`、`idx_contacts_last_message_at`、`idx_contacts_status`、`idx_messages_contact_id`、`idx_messages_contact_id_desc` 等。
  - 戰情用的條件（brand_id、needs_human、assigned_agent_id、status、last_message_at、closed_at、created_at）皆可落在現有索引或單欄索引上；**未新增索引**，僅改查詢方式為 COUNT。

### 已實作修改

- 如上所述：戰情數字改為 `getManagerStatsCounts`、`getAgentStatsCounts`、`getAgentPendingReplyCount`，全部為 COUNT（與必要子查詢），不再用 `getContacts` 全表載入。

| 檔案 | 變更摘要 |
|------|----------|

## 建議後續可做（未實作）

1. **前端「載入更多」**  
   左側列表可改為先只取 `limit=100`，底部加「載入更多」按鈕，點擊時以 `offset=100,200,...` 再打 GET /api/contacts 並 append 至列表（或合併進同一 queryKey 的 cache）。

2. **戰情僅依 SSE 更新**  
   若 SSE 已有 `contacts_updated` 等事件，可考慮戰情數字不設 refetchInterval，僅在 SSE 收到時 invalidate/refetch，進一步減少輪詢。

3. **staleTime 與 invalidate 範圍**  
   文件 `performance-review-switch-slowness.md` 已建議：聯絡人 query 的 staleTime 可視情況拉長；invalidate 時可收斂為只 invalidate 當前 view 的 queryKey，避免全站 refetch。

---

## 小結

- **列表 API**：已有 limit；已補 **offset 分頁**；戰情改為 **COUNT 專用 API**，不再全表載入。
- **前端列表**：已用 **虛擬列表**，無需改動。
- **輪詢**：聯絡人由 **5 秒且背景輪詢** 改為 **20 秒且背景不輪詢**。
- **DB**：無 N+1；索引已足夠；戰情改為 COUNT 後，負載與卡頓預期明顯改善。

以上修改在不改動既有業務邏輯與權限的前提下，針對「列表與戰情在 2000+ 案件時的效能瓶頸」做了對應重構，並保留後續可分頁、可再降輪詢的空間。
