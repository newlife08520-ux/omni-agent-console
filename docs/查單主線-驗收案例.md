# 查單主線重構 — 正式驗收案例

以下案例需在**實際環境**或**整合測試**中逐筆驗證，回報 PASS / FAIL。

---

## 1. 圖裡有品牌訂單頁 + 明確訂單編號 + 綠界字樣

**步驟**：上傳一張圖片，內容同時包含品牌訂單成立頁、明確訂單編號（如 ESC20815）、以及綠界/ECPay 字樣。

**預期**：系統先用 order_id 查單（extractOrderInfoFromImage → unifiedLookupById），回覆「已收到您的圖片，幫您查到這筆訂單：」+ 完整訂單資訊；**不可**先回「請補充訂單編號」或誤判為付款截圖模板。

**實作位置**：`server/routes.ts` — `handleImageVisionFirst` 開頭，先 `extractOrderInfoFromImage`，若有合法 order_id 則 `unifiedLookupById`，成功則設 active_order_context 並回傳訂單摘要。

| 結果 | PASS / FAIL |
|------|-------------|
| 實際驗證 | （待測） |

## 2. 成功查到訂單後，客人問「什麼時候到貨」

**步驟**：先以訂單編號或圖片成功查單（已建立 active_order_context），下一則客人發「什麼時候到貨」。

**預期**：AI 依 **active_order_context** 回答（訂單狀態、出貨時間、或「約 7–20 工作天」），**不可**直接回「這邊先幫您轉接真人專員」或呼叫 transfer_to_human。

**實作位置**：`server/routes.ts` — order_lookup 的 system prompt 注入「當前訂單上下文」+ 本輪禁止查單/轉人工。

| 結果 | PASS / FAIL |
|------|-------------|
| 實際驗證 | （待測） |

**預期**：依 **active_order_context** 的 payment_status / 訂單狀態直接回答（成功/待付款/失敗），不轉人工、不重問資料。

## 4. 成功查到訂單後，客人回「好的」（或任何可收尾情境）

**步驟**：先成功查單並回覆後，客人回「好的」或類似道別／已滿意的短句。

**預期**：AI **自行判斷**感覺能收尾了就自然收尾（簡短一句），**不再**主動開新題、不追問、不給多選項。收尾與否由 AI 依情境判斷，不限於客人是否說「好的」「謝謝」。

**實作位置**：`server/routes.ts` — prompt「收尾判斷」：由 AI 自己判斷何時可收尾，任何情況只要判斷能收尾就收尾。

| 結果 | PASS / FAIL |
|------|-------------|
| 實際驗證 | （待測） |

**實作位置**：`client/src/pages/chat.tsx` — `getPaymentStatusLabel`、訂單卡片「付款狀態」badge；AI 依 active_order_context 的 payment_status 回答。

## 7. 圖片 + 文字 + 已查到單，後續問題都接著同一筆單

**預期**：後續每一問皆依 **同一筆** active_order_context 回答，不重問訂單編號、不重跑查單入口、不轉人工（除非客人明確要求轉專人或查另一筆）。

## 8. 先查到 A 單，再貼 B 單編號

**步驟**：先以訂單編號或圖片查到訂單 A（active_order_context = A），下一則客人貼出或輸入訂單 B 的編號。

**預期**：系統以 B 編號查單，**active order 正確切換為 B**；回覆內容為 B 單資訊，**不可**沿用 A 單。

**實作要點**：客人訊息含新訂單編號時不會被視為「訂單延伸問句」，不注入當前 A 單；AI 呼叫 lookup_order_by_id(B) 成功後 setActiveOrderContext 會覆寫為 B。

| 結果 | PASS / FAIL |
|------|-------------|
| 實際驗證 | （待測） |

## 9. 查到單後，客人說「不是這筆」

**步驟**：先成功查單（已建立 active_order_context），客人回「不是這筆」或「這筆不對」。

**預期**：**解除／切換 active order**（clearActiveOrderContext），後續可重新引導訂單編號或商品+手機；**不可**繼續沿用錯單回答。

**實作位置**：`server/routes.ts` — order_lookup 區塊內，`/不是這筆/` 時 clearActiveOrderContext(contact.id)。

## 10. 查到單後，客人只問「有出貨嗎」

## 回報補充（必填）

### 1. active order 什麼情況會清除／切換

| 情境 | 行為 |
|------|------|
| **清除** | 客人說「另一筆」「新訂單」「別筆」「其他訂單」「換一筆」「我要查別的」「**不是這筆**」時，`clearActiveOrderContext(contact.id)`，後續可重新引導或查另一筆。 |
| **切換** | 客人提供**新訂單編號**並經 `lookup_order_by_id` 查詢成功，或提供產品+手機經 `lookup_order_by_product_and_phone` 僅回傳一筆時，`setActiveOrderContext` 會**覆寫**為該筆，即 active order 切換為新查到的單。 |
| **維持** | 客人問「什麼時候到貨」「有出貨嗎」「付款有成功嗎」等延伸問句且未說換筆/不是這筆時，不清除、不切換，沿用當前 active 回答。 |

---

### 2. 圖片抓錯 order_id 時的保底處理

| 項目 | 說明 |
|------|------|
| **流程** | `handleImageVisionFirst` 先呼叫 `extractOrderInfoFromImage` 取 order_id → 有則 `unifiedLookupById`。若**查不到**（非該品牌、編號錯誤、或 vision 抽到的是金流序號等），會 **catch 後繼續走原本 vision 意圖流程**，不會卡死。 |
| **保底** | 抽到錯的編號導致 lookup 失敗時：不寫入 active_order_context，不回「查到這筆訂單」；同一輪可依圖片內容由後續 vision/LLM 判讀（例如引導「請提供訂單編號」或改用文字查單）。 |
| **改進空間** | 若品牌訂單編號有固定前綴（如 ESC），可在 `extractOrderInfoFromImage` 或呼叫前過濾「非訂單格式」；目前依 vision 回傳 + 長度 ≥5 與合法字元即嘗試查單，查不到即 fallback。 |

---

### 3. 查到單後，哪些情況回完整摘要，哪些情況只回局部答案

| 情境 | 預期行為 |
|------|----------|
| **回完整摘要** | ① **剛查到單的當輪**：工具回傳 one_page_summary 時，**必須在回覆中完整貼給客戶**（訂單編號、收件人、電話、下單時間、付款方式、金額、配送、物流單號、地址/門市、商品、狀態等）。② **多筆訂單**：須逐筆列出，不可只列一筆。 |
| **只回局部答案** | ① **已有 active 且客人只問單一重點**（如「有出貨嗎」「什麼時候到」「付款成功了嗎」）：**只回答該重點**，勿每輪重貼完整訂單摘要。② 客人已看過完整摘要、後續追問細節時：依問的項目回答即可。 |
