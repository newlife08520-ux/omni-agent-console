# 緊急案件：觸發條件與分配說明

## 為什麼「感覺太容易就列入」？

目前「緊急」的判定是**多條件任一符合即成立**，且**每則用戶訊息都會重新計算標籤與優先級**，所以很容易被列入。

---

## 一、緊急案件是怎麼「觸發」的？（後端 `isUrgentContact`）

程式位置：`server/routes.ts` 的 `isUrgentContact`（約 L2570），以及 API 回傳給前端的 `is_urgent`。

任一條件成立，該聯絡人就會在 UI 上顯示為**緊急**（紅點／緊急案件篩選／戰情數字）：

| 條件 | 說明 | 為何容易觸發 |
|------|------|----------------|
| **1. status === "high_risk"** | 系統已標記為高風險 | 僅當用戶訊息含有**法務／公關關鍵字**（如 消保官、提告、媒體、爆料、詐騙）時，由 `detectHighRisk` 短路並寫入 `high_risk`。門檻高，不常觸發。 |
| **2. case_priority ≤ 2** | 案件優先級為 1 或 2 | **每則訊息**都會跑 `suggestTagsFromContent` + `computeCasePriority`，並寫入 `case_priority`。見下方「case_priority 怎麼被設成 1／2」。 |
| **3. 標籤在 URGENT_TAGS** | 聯絡人標籤含有任一個 | `URGENT_TAGS = ["客訴", "退款", "轉主管", "緊急案件"]`。**「緊急案件」是其中一個標籤**，快速選取一點就列入；另「客訴」「退款」「轉主管」若被加上也會變緊急。 |
| **4. VIP + 逾時** | VIP 且最後一則為客戶且超過 1 小時未回 | 只要該聯絡人為 VIP，且最後一則來自客戶、超過 1 小時，就視為緊急。門檻較寬。 |
| **5. SLA 逾期** | `response_sla_deadline_at` 已過 | 有設 SLA 且逾期的案件會顯示為緊急。 |

所以「太容易」主要來自：**2（case_priority）** 和 **3（標籤，尤其是手動點「緊急案件」）**。

---

## 二、case_priority 怎麼被設成 1／2？（每則訊息都會跑）

程式位置：`server/routes.ts` 約 L3920–3927，每則 Webhook 進來都會執行：

1. **suggestTagsFromContent(userMessage, currentTags)**  
   依**本則用戶訊息**建議要加的標籤（只追加、不刪除既有）：
   - 含「訂單|查單|出貨|物流」→ 建議「訂單查詢」
   - 含「缺貨|欠貨|補貨|到貨」→ 建議「缺貨 / 欠貨」
   - 含「退款|退貨|取消」→ 建議「**退款 / 取消**」

2. **computeCasePriority(intentLevel, tags)**（`server/intent-and-order.ts`）  
   依**意圖強度 + 標籤**算出數字優先級（1 最高）：
   - 標籤有「出貨延遲」「缺貨 / 欠貨」「**退款 / 取消**」「客訴」→ **1**
   - 標籤有「訂單查詢」且 intentLevel === "high" → **1**
   - intentLevel === "high" → **2**
   - 標籤有「待人工接手」「高意圖」→ **2**
   - 其餘 3 / 4 / 5…

3. **storage.updateContactCasePriority(contact.id, priority)**  
   把算出來的數字寫進 DB。

因此：

- 只要用戶**隨便一句話裡出現「退款」「退貨」「取消」**，就會被建議「退款 / 取消」→ `computeCasePriority` 回傳 **1** → 該聯絡人立刻變成**緊急**（因為 `case_priority <= 2`）。
- 若用戶說「我要查訂單」等，被建議「訂單查詢」且意圖被判為 high，也會得到 priority **1**，同樣變緊急。

這就是「感覺太容易就列入」的**主要來源**：**門檻低、且每則訊息都重算一次**。

## 三、緊急案件是怎麼「分配」的？

- **「緊急」只影響「顯示」與「篩選」**：左側列表緊急篩選、戰情數字、列表排序（緊急排前面）等。
- **實際分配**仍由既有邏輯決定：
  - 待分配池：`needs_human === 1` 且 `assigned_agent_id` 為空。
  - 分配方式：主管手動分配 或 自動分配（`assignment.assignCase`），**沒有**「緊急案件優先分配」的獨立邏輯。
- 因此會出現：畫面上是「緊急案件」、但同時顯示「目前為假日非服務時間…下個工作日為您處理」——**緊急只代表標記/排序，不改變假日不派案、或全員忙碌時不派案**。

## 四、若希望「不要那麼容易就列入緊急」可怎麼調？

（以下為建議方向，實作時需依貴司流程再微調。）

1. **收緊 case_priority 觸發**
   - 例如：只有當標籤同時有「客訴」或「退款/取消」**且**意圖為 high 時，才設為 1；  
     或「退款/取消」改為只設 2，不設 1。
   - 或：**不要每則訊息都寫入 case_priority**，改為僅在「狀態變更」（如轉人工、或標籤從無到有）時才重算並寫入。

2. **區分「緊急案件」標籤與「顯示為緊急」**
   - 例如：手動標「緊急案件」仍可顯示為緊急；  
     但**僅**因「退款/取消」「訂單查詢+高意圖」等自動建議而得到 priority 1／2 的，**不要**算進「緊急案件」戰情／篩選，或改用另一種視覺（例如「優先」而非「緊急」）。

3. **URGENT_TAGS 與標籤建議**
   - 若希望「退款」不要一加標就變緊急，可考慮從 `URGENT_TAGS` 拿掉「退款」，或改為僅「客訴」「轉主管」「緊急案件」才算緊急。
   - 或「緊急案件」僅能手動加，不透過其他標籤自動帶入。

4. **VIP + 逾時**
   - 若覺得 VIP 逾時 1 小時就變緊急太寬，可改為逾時更長（例如 2 小時）才計入緊急，或從緊急條件中拿掉此條。

## 五、程式位置速查

| 項目 | 檔案 | 約略位置 |
|------|------|----------|
| 緊急判定 isUrgentContact、URGENT_TAGS | `server/routes.ts` | L2567–2585 |
| 法務/公關 high_risk 短路 detectHighRisk | `server/routes.ts` | L190–205, L3792–3796 |
| 每則訊息建議標籤 + 寫入 case_priority | `server/routes.ts` | L3920–3927 |
| computeCasePriority、suggestTagsFromContent | `server/intent-and-order.ts` | L89–111 |
| 分配邏輯（無緊急優先） | `server/assignment.ts` | getEligibleAgents / assignCase |

若要調整「多容易才算緊急」，可從 **`computeCasePriority` 回傳 1／2 的條件** 與 **`URGENT_TAGS`** 兩處著手，再視需要加上「僅在特定時機寫入 case_priority」的條件。
