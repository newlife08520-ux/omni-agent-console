# 品牌與渠道 — 常見混亂與排除

本文說明「切換品牌卻看到別品牌的 LINE 訊息」「刪除品牌失敗」「新增渠道搞混品牌」的原因與處理方式。

---

## 一、為什麼切換天鷹座卻看到私藏生活的 LINE 訊息？

### 原因：LINE 用「Bot ID = destination」決定訊息歸哪一品牌

系統收到 LINE Webhook 時，會用 **LINE 傳來的 `destination`** 去資料庫查「哪一個渠道的 Bot ID 等於這個 destination」。  
查到的渠道屬於哪個品牌，**新訊息就會歸到那個品牌**。

- 若某個 LINE 渠道的 **Bot ID 填錯**（例如填成 `U` 開頭的 **User ID**），則 LINE 實際送來的 `destination` 是 **Channel ID**（通常不是 U 開頭），與資料庫裡的 Bot ID 對不起來 → **查不到渠道**。
- 此時該則訊息無法歸到任何品牌（或會用全域 token 處理但 contact 的 brand_id 可能為 null），結果可能是：
  - 訊息只出現在「全部」、在個別品牌下看不到；或
  - 若兩個品牌曾共用／填錯同一個 Bot ID，就可能出現 A 品牌畫面看到 B 品牌訊息的錯亂。

### 正確做法

1. **到 LINE Developers 後台**：該 Bot 的 **Channel ID**（或 Webhook 設定頁會看到的 ID），才是 LINE 送進 Webhook 時的 `destination`。
2. **到 Railway 日誌**：收到 LINE 訊息時會印出 `[WEBHOOK] destination: xxxxx`，**這個值**就是該 LINE Bot 的識別碼。
3. **在「品牌與渠道」**：編輯該 LINE 渠道，把 **Bot ID** 改成上述 **destination 的值**（不要填 `U` 開頭的 User ID），儲存。
4. **一品牌一 Bot**：每個品牌若有自己的 LINE 官方帳號，就各自建一個渠道，且每個渠道的 Bot ID 都要填**該 Bot 的 destination**，彼此不要填錯或重複。

完成後，新進的 LINE 訊息就會正確歸到對應品牌，切換天鷹座就只會看到天鷹座的對話。

### 從 Railway 日誌複製 destination 的步驟

1. 登入 **Railway** 專案，點進部署的服務（Omni-Agent-Console server）。
2. 開啟 **Logs**（日誌）分頁，保持即時串流。
3. 用手機或模擬器對「要設定的那個 LINE 官方帳號」**傳一則訊息**（例如傳「hi」）。
4. 在日誌裡搜尋 `[WEBHOOK] destination:` 或 `destination:`，會看到類似：
   ```text
   [WEBHOOK] destination: Ue2de0380dea73fdf87a50548a8a4f00e55ad
   ```
   或（依 LINE 版本可能是 Channel ID 格式）：
   ```text
   [WEBHOOK] destination: 1234567890
   ```
5. **複製冒號後方那一整串**（不含空格），即為該 LINE Bot 的識別碼。
6. 到後台 **品牌與渠道** → 選取對應品牌 → 編輯該 LINE 渠道 → 把 **Bot ID** 欄位貼上這串值 → **儲存**。

注意：若日誌裡出現的是 `U` 開頭，LINE 有時會送 User ID 當 destination；若你有多個 LINE Bot，請對「要設定的那一個」發訊，才不會抄錯。若同一 Webhook URL 被多個 LINE 帳號使用，每個帳號發一則訊再分別從日誌抄下對應的 destination，各自填進對應品牌渠道。

## 一之一、「私藏生活綁對卻沒訊息，天鷹座一堆訊息卻都是私藏的」— 一次搞懂

### 你現在的情況（對應到系統怎麼運作）

- **私藏生活**：你有建渠道、也覺得「綁的是對的」，但該品牌下**沒有訊息**。
- **天鷹座**：該品牌下**一堆訊息**，但其實那些都是**私藏生活**的客人傳的。

代表：**所有從「私藏生活那個 LINE 帳號」進來的訊息，目前都被系統歸到天鷹座**。

### 為什麼會這樣？

系統**只靠一個東西**決定「這則 LINE 訊息要歸哪一個品牌」：  
**Webhook 裡的 `destination` = 某個渠道的 Bot ID** → 該渠道所屬的品牌就會收到這則訊息。

所以會變成你現在這樣，只可能是下面其中一種（或混合）：

1. **兩個品牌的 Bot ID 填反了**  
   - 私藏生活渠道的 Bot ID 填成「天鷹座那支 LINE 的 destination」。  
   - 天鷹座渠道的 Bot ID 填成「私藏生活那支 LINE 的 destination」。  
   - 結果：私藏生活客人傳的訊息，destination 對到天鷹座渠道 → 全部進天鷹座；私藏生活渠道永遠對不到 → 沒訊息。

2. **兩個品牌共用同一個 LINE 官方帳號**  
   - 只有一個 LINE Bot、一個 destination，但你有兩個品牌、兩個渠道。  
   - 系統只能把這個 destination 對到「其中一個」渠道（例如先建的天鷹座），所以所有訊息都進那一個品牌，另一個品牌永遠沒訊息。

3. **其中一個（或兩個）Bot ID 填錯**  
   - 例如填成 User ID、或抄錯，導致「私藏生活 LINE 的 destination」只對得到天鷹座那一個渠道，私藏生活渠道對不到。

所以：**私藏生活「綁對」是指你在畫面上有幫私藏生活建渠道，但「實際送進來的 destination」沒有對到這個渠道，而是對到天鷹座的渠道。**

### 你要怎麼修（對應你兩支 LINE）

前提：**一個品牌一支 LINE 官方帳號**，兩支要分開。

| 步驟 | 做什麼 |
|------|--------|
| 1 | 用手機對 **私藏生活** 的 LINE 傳一則「hi」。 |
| 2 | 立刻到 **Railway Logs** 找 `[WEBHOOK] destination: xxxxx`，**複製那串 xxxxx**（例如 `Ub3fa547cb3ec7c7975e07ab359cb`）。 |
| 3 | 後台 → **品牌與渠道** → 選 **私藏生活** → 編輯 **私藏生活 LINE** 渠道 → **Bot ID 貼上剛複製的那串** → 儲存。 |
| 4 | 再用手機對 **天鷹座** 的 LINE 傳一則「hi」。 |
| 5 | 再到 Railway Logs 找這次的 `[WEBHOOK] destination: xxxxx`，**複製這串**（應該會和私藏生活的不一樣）。 |
| 6 | 後台 → 選 **天鷹座** → 編輯 **天鷹座 LINE** 渠道 → **Bot ID 貼上天鷹座這串** → 儲存。 |

這樣之後：

- 私藏生活客人傳的訊息 → destination 對到私藏生活渠道 → 訊息出現在 **私藏生活**。  
- 天鷹座客人傳的訊息 → destination 對到天鷹座渠道 → 訊息出現在 **天鷹座**。

**已經進錯的舊訊息**：系統是依「當時的 destination → 當時的 Bot ID」歸檔的，不會自動搬移。之後新訊息會歸對；若要整理舊案件，可再考慮手動改聯絡人所屬品牌或從「全部」裡處理。

**若你只有一支 LINE 要給兩個品牌用**：系統無法依「品牌」自動分開，只能都進同一個品牌（或「全部」）。要分開就必須有兩支 LINE 官方帳號，並照上面各自填對 Bot ID。

### 系統已做的兩項修正（減少錯亂）

1. **新訊息會依「目前匹配到的渠道」更新聯絡人歸屬**  
   以前：聯絡人一旦被歸到某品牌就不會改，所以第一次錯歸就永遠錯。  
   現在：每次 LINE 訊息進來且系統匹配到某個渠道時，會把該聯絡人的品牌／渠道更新為「這次匹配到的」，所以**只要把 Bot ID 改對，之後新訊息就會自動歸到正確品牌**，不必手動搬聯絡人。

2. **批次修正舊聯絡人**  
   若已經有一堆訊息錯歸在 A 品牌，但其實都是從「B 品牌的渠道」進來的，可以一次拉回：  
   到 **品牌與渠道** 頁面下方 **「批次修正聯絡人歸屬」**：選擇「該渠道」→ 選擇「要改歸到的品牌」→ 按 **執行**。  
   會把「該渠道下所有聯絡人」的 brand_id 改為你選的品牌，之後他們就會出現在正確品牌的即時客服列表。

## 二、為什麼刪除品牌會失敗（刪除失敗）？

### 原因（已修復）

先前刪除時會先問 **Redis** 有沒有這個品牌；若 Redis 裡沒有（例如 Redis 曾清空、或品牌只存在 SQLite），就回傳失敗，**連 SQLite 都不會刪**，所以畫面上會一直顯示「刪除失敗」。

### 修復後邏輯

現在改為**以 SQLite 為準**：

1. 先從 SQLite 刪除該品牌的渠道與品牌。
2. 若系統有啟用 Redis，再盡量同步從 Redis 移除（同步失敗不影響刪除結果）。

因此只要後台「品牌列表」裡看得到該品牌，刪除應會成功。若仍失敗，請看瀏覽器開發者工具 Network 裡該 DELETE 請求的回應內容（例如 404「品牌不存在」），或檢查是否有其他錯誤訊息。

## 三、新增渠道時會不會加錯品牌？

### 邏輯說明

- 「**XXX 的渠道**」區塊是**依目前選取的品牌**顯示的。
- 按「**新增渠道**」時，會對 **目前選取的品牌** 建立渠道（API：`POST /api/brands/{selectedBrandId}/channels`）。
- 新增渠道的彈窗裡有一行：「**此渠道將隸屬於：品牌名稱**」，可再確認一次。

### 建議操作

1. 先點品牌列表該品牌右側的「**選取**」，確認左側／上方已顯示該品牌為目前工作區。
2. 再按「**新增渠道**」，並在彈窗中確認「此渠道將隸屬於：OOO」是否正確。
3. LINE 的 **Bot ID** 務必填 **Railway 日誌中的 destination**，不要填 U 開頭的 User ID。

這樣就不會把渠道加錯品牌。

## 四、快速檢查清單

| 狀況 | 請檢查 |
|------|--------|
| 切到 A 品牌卻看到 B 品牌的 LINE 訊息 | 各 LINE 渠道的 Bot ID 是否為 Railway 日誌的 `destination`；是否有一 Bot 被多品牌共用或填錯。 |
| 刪除品牌一直顯示刪除失敗 | 已改為以 SQLite 為準，理論上可刪；若仍失敗請看 API 回應。 |
| 新增渠道怕加錯品牌 | 先「選取」要的品牌，再新增渠道，並看彈窗「此渠道將隸屬於：XXX」。 |
| LINE 訊息只在「全部」、品牌下沒有 | 該 LINE 渠道的 Bot ID 很可能填錯（例如填成 U 開頭），請改為 destination 後儲存。 |
| 不知道 destination 從哪抄 | 見上方「從 Railway 日誌複製 destination 的步驟」：對該 LINE 發一則訊 → 開 Railway Logs → 搜尋 `destination:` → 複製冒號後的值貼到 Bot ID。 |

---

*若你使用 Redis，刪除品牌後 Redis 會盡量同步；若未使用 Redis，僅以 SQLite 為準即可。*
