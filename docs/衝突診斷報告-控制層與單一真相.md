## 二、A. 目前到底有幾層在同時控制回覆？

依程式執行順序列出，**同一則 user message 會依序經過**：

| 層級 | 名稱 | 位置 | 作用 |
|------|------|------|------|
| **L0** | Webhook / Channel mapping | `routes.ts` LINE/FB webhook | 依 `destination`/bot_id 找 channel；無 match 則 fail-closed，不進後續。Token/secret 決定能否調 LINE/FB API。 |
| **L1** | 前置 Gate（靜音/已轉/高風險） | `autoReplyWithAI` 開頭 | `awaiting_human` / `high_risk` / `needs_human` / `isAiMuted` → 直接 return，不進 LLM。 |
| **L2** | 高風險關鍵字（routes 內） | `detectHighRisk(userMessage)` | 命中 `HIGH_RISK_KEYWORDS`（含「爛」）→ 設 high_risk、needs_human、寫 system 訊息、**return**，不進後續。 |
| **L3** | 安全確認分流（Safe After Sale） | `classifyMessageForSafeAfterSale` | 詐騙/他平台/待確認來源 → 回模板、可 suggest_human、**return**。 |
| **L4** | 圖片＋極短/模糊文字 | `hasRecentImageFromUser` + `isShortOrAmbiguousImageCaption` | 回「已收到您的圖片～…」類模板、**return**（已加高風險防呆：本則高風險則不進此分支）。 |
| **L5** | Conversation State | `resolveConversationState` | 產出 `primary_intent`、`return_reason_type`、`needs_human`、`human_reason`、`return_stage` 等，**不直接決定回覆內容**，供下一層用。 |
| **L6** | Reply Plan Builder | `buildReplyPlan` | 依 state 產出**單一** `plan.mode` 與 `must_not_include` / `should_include`。 |
| **L7** | 固定模板 / Short-circuit | `plan.mode === "return_form_first"` | 固定退換貨＋表單文案、寫 return_stage、**return**，不進 LLM。 |
| **L8** | Prompt / Persona / handoffBlock | `getEnrichedSystemPrompt` + 本輪追加 | 全域 + 品牌 + handoffBlock + 人工時段 + 目錄 + 知識庫 + 圖片素材；本輪再追加 F2、handoff、久候型等。 |
| **L9** | LLM Generator | OpenAI chat completions | 依 prompt 與 tools 生成回覆；**仍可自選是否呼叫工具、多輪 tool call**，未強制「本輪只做一件事」。 |
| **L10** | Tool call / transfer_to_human | executeToolCall | 查單、轉人工、發圖等；**由 LLM 決定是否呼叫**，無強制「mode=handoff 必呼叫 transfer」。 |

**衝突要點**：

- **L2 高風險**：單字「爛」即觸發，與需求「情緒差 vs 法律/公關風險分開」不符。
- **L5 意圖**：用 `combined = [text, ...recentUserMessages]`，**未以本則為準做 correction override**，易出現「上一句退貨、本句說錯要查出貨」仍被當退貨。
- **L6 雖單一 mode**：但 **L9 未硬性約束**「本輪只能做 plan.mode 對應行為」，LLM 仍可同時查單+問類別+提其他平台。
- **L8 handoffBlock**：與 L6/L7 規則重疊且部分矛盾（例如「不可一開始就丟表單」vs 商品問題型固定表單），易造成 prompt 與程式雙源頭。

## 四、C. 哪些是「AI 邏輯錯」vs「管道配置錯」？

**管道／配置錯（需獨立排查與修正）**：

| 現象 | 可能原因 | 排查方式 |
|------|----------|----------|
| 訊息只出現在「全部」、brand_id 為 null | destination 與 DB bot_id 不一致 → NO MATCH | 查 log `[WEBHOOK] NO MATCH for bot_id`；對齊 bot_id 與 destination |
| 頭像/姓名預設、媒體下載失敗 | 該 channel 無 token 或 token 錯誤（401） | 查 `[LINE API Error] Channel ID`、Token 防呆 log；確認該渠道 access_token |
| 簽章 403 | channel_secret 與 LINE 後台不一致 | 查 `SIGNATURE MISMATCH`；對齊 Channel Secret |
| Token available: false | 無 matchedChannel 或 access_token 空 | 同上，屬 channel 設定問題 |

**AI／邏輯錯（state / plan / prompt / 高風險定義）**：

| 現象 | 可能原因 |
|------|----------|
| 該查單時亂問類別 | 無「order_lookup 禁止問分類」的硬約束；LLM 自由發揮 |
| 該轉人工時還在拖 | human_request 未涵蓋「人呢」等說法；或 needs_human 已設但 prompt 仍讓 LLM 收資料 |
| 該簡短卻長篇 | 無字數/段數/問題數的輸出契約，僅 prompt 寫「簡潔」 |
| 包包跳巴斯克 | 無商品範圍鎖定與「禁止跨類」 |
| 離題陪聊（晚餐吃什麼） | 無 off_topic_guard 與邊界回覆 |
| 高風險過敏（爛就整段變形） | HIGH_RISK_KEYWORDS 含情緒字，未拆 legal vs frustrated |
| 顧客糾正後仍走舊流程 | 意圖用 combined，未做「本則明確糾正覆蓋」 |

**結論**：  
- 先區分：**收不到、對錯 channel、token/secret 錯** → 屬管道與設定，修 mapping/token/secret 與 fail-closed 邏輯。  
- **收得到但回錯、問錯、拖轉人工、跨類、離題、高風險誤判** → 屬控制層與單一真相，修 state/plan/mode/高風險分級/prompt 契約。

## 五、單一真相優先級（建議落地順序）

| 優先級 | 內容 | 說明 |
|--------|------|------|
| **P0** | 渠道 / webhook / bot mapping 正確性 | destination=bot_id、token/secret 正確、fail-closed 不進正常流程、log 可追查 |
| **P1** | 硬性前置 Gate | awaiting_human / high_risk(真) / muted → 不進 LLM；高風險拆級（legal vs frustrated） |
| **P2** | Conversation State | 單一來源；補 correction override、必要時 last_resolved_intent / product_scope_locked |
| **P3** | Reply Plan（本輪唯一 mode） | 已存在；補不足 mode（off_topic_guard 等），嚴格一輪一 mode |
| **P4** | 固定模板 / deterministic | 僅 return_form_first 等短路；其餘由 mode 契約約束 |
| **P5** | LLM 生成語氣 | 在 mode 下組話術；不得推翻 plan.mode |
| **P6** | Persona 風格 | 只影響用語，不決定流程 |

**原則**：人格不決定流程；prompt 不推翻 reply plan；LLM 不自由切換 mode；文件與程式一致，以程式為單一來源。

## 六、同一輪只能一個 mode（現況與缺口）

**現有 mode（reply-plan-builder）**：  
answer_directly、ask_one_question、order_lookup、aftersales_comfort_first、return_stage_1、return_form_first、handoff、fallback_unknown、idle_closure、invite_rating。

**缺口**：

- 無 **handoff_now**（明確要真人立刻轉，不再問）。
- 無 **high_risk_handoff**（真法務/公關/詐騙，與 frustrated 分開）。
- 無 **return_insist_handoff**（可併入 handoff 或獨立）。
- 無 **off_topic_guard**（品牌外問題短邊界）。
- 無 **product_qa** / **product_matching** 細分（目前多為 answer_directly）。
- 無 **clarify_missing_info** / **unsupported_request** 等。
- **商品範圍鎖定**（包包不跳蛋糕）未落在 mode 或 state，僅靠目錄與知識庫，易跨類。

**規則**：一輪只進一個 mode；由 state + 優先級決定；LLM 只在該 mode 下把話講自然，不得同輪查單+問退貨原因+提其他平台+轉人工+問類別。

## 七、高風險分級（現況與建議）

**現況**：`HIGH_RISK_KEYWORDS` 含「爛」「垃圾」「很煩」等，單字即觸發 → needs_human + high_risk + system 訊息 + return。

**問題**：與需求「爛、很煩、很慢、不爽 → frustrated_only；提告/消保/檢舉/詐騙/公開爆料/媒體/法律 → 真 high_risk」不符。

**建議**：

- **legal_risk_keywords**：投訴、消保、檢舉、詐騙、提告、媒體、爆料、法律等 → high_risk_handoff，不進 LLM。
- **frustrated_only_keywords**：爛、很煩、很慢、不爽等 → 僅標記 frustrated 或 emotion，不觸發 high_risk，可進 LLM 或 handoff 依情境。

## 八、Webhook / Channel / Bot 對應摘要

- **LINE**：`destination` 與 `storage.getChannelByBotId(destination)` 對應；無 match 則不 fallback 全域 token，不進行自動回覆（fail-closed）；日誌有 NO MATCH、Token available、SIGNATURE MISMATCH。
- **Token/Secret**：來自 matchedChannel；不應在品牌專屬 API 使用全域 `LINE_CHANNEL_ACCESS_TOKEN`。
- **儲存前**：POST/PUT channels 已對 LINE 做 Token 驗證（GET bot/info），無效則 400。

**獨立排查建議**：  
對「收不到、頭像/媒體失敗、403」的案例，先查 log 是否為 NO MATCH / Token false / SIGNATURE MISMATCH；再查 DB 該品牌 channel 的 bot_id、access_token、channel_secret 是否與 LINE 後台一致。

## 十一、建議實作順序（不在此文件改程式）

1. **高風險分級**：拆 legal_risk vs frustrated_only；僅 legal 觸發 L2 短路。
2. **意圖 correction override**：本則若明顯糾正前意圖（如「說錯，我要查…」），以本則為主重算 intent，或清掉前輪暫存意圖。
3. **human_request 補 pattern**：「人呢」「我要人工」「能轉人工嗎」等。
4. **mode 擴充**：handoff_now、high_risk_handoff、off_topic_guard 等；嚴格一輪一 mode。
5. **order_lookup 契約**：在 prompt 或 mode 契約中禁止「問分類問卷」、禁止多餘欄位。
6. **商品範圍鎖定**：state 或 contact 層記錄已鎖定品類；reply 或 prompt 禁止跨類。
7. **回覆長度契約**：首輪字數/段數/問題數上限，寫在程式或 mode 契約，不只寫「請簡潔」。
8. **Prompt 拆層**：persona / style / policy / mode-specific contract；mode 由 state+plan 決定，prompt 不奪權。
9. **Webhook/Channel 可觀測性**：錯誤分類碼、哪個 brand/channel/destination 出錯，方便與「業務邏輯錯」分開。
