# 指派客服卡住／CHECK constraint 報錯 - 修復說明

## 1. 真正 bug 原因

- **直接原因**：資料庫 `contacts` 表的 `status` 欄位有 CHECK 約束，只允許部分舊狀態（如 `pending`, `processing`, `resolved`, `ai_handling`, `awaiting_human`, `high_risk`, `closed`），**不包含 `assigned`**。手動／自動指派時，後端會把該筆對話的 `status` 更新為 `assigned`，寫入時觸發 CHECK constraint failed，導致 API 丟錯、前端收到錯誤後 modal 沒關、loading 沒正確結束、又只顯示資料庫錯誤原文，體驗很差。
- **間接原因**：  
  - 遷移「擴充 contacts.status 支援案件流程狀態」可能沒在現有 DB 跑成功（例如 regex 沒對到 schema 格式）。  
  - 前後端沒有共用的「合法 status 清單」，無法在寫入前擋掉非法值。  
  - 指派／改派 API 未捕獲 DB 例外，錯誤直接拋出。  
  - 前端在失敗時沒有強制關閉 modal、沒有把 constraint 錯誤轉成人類可讀訊息。

---

## 2. 改了哪些檔案

| 檔案 | 修改內容 |
|------|----------|
| `shared/schema.ts` | 新增 `CONTACT_STATUS_ALLOWED` 常數（與 DB CHECK 一致），供後端驗證與文件用。 |
| `server/db.ts` | 遷移 regex 改為可匹配多行 schema（`[\s\S]*?`），確保「補強 status CHECK」能正確替換並加入 `assigned` 等狀態。 |
| `server/storage.ts` | `updateContactStatus` 寫入前檢查 `status` 是否在 `CONTACT_STATUS_ALLOWED`，非法則拋錯，避免寫入 DB 觸發 CHECK。 |
| `server/routes.ts` | **assign**：整段指派邏輯包在 try/catch，捕獲到 CHECK/constraint 類錯誤時回傳 500 與友善訊息。**reassign**：同樣加 try/catch，constraint 時回傳 500 與友善訊息。 |
| `client/src/pages/chat.tsx` | **手動指派**：失敗時一律關閉指派 modal、清空所選客服、結束 loading；解析 API 錯誤 body 顯示後端訊息，若為 CHECK/constraint 則顯示「請重新整理頁面後再試」。**自動分配**：失敗時關閉 modal、顯示後端或 constraint 友善訊息。**改派**：失敗時關閉改派 modal、結束 loading、顯示友善或 constraint 訊息。 |

---

## 3. 每個檔案改了什麼（摘要）

- **shared/schema.ts**  
  - 在 `ContactStatus` 型別下方新增 `CONTACT_STATUS_ALLOWED: ContactStatus[]`，列出與 DB 一致的 14 個合法狀態（含 `assigned`, `waiting_customer` 等）。

- **server/db.ts**  
  - `migrateContactStatusCaseFlow`、`ensureContactStatusIncludesAssigned` 中，用來匹配 `CHECK(status IN (...))` 的 regex 改為可匹配多行：`/CHECK\s*\(\s*status\s+IN\s*\([\s\S]*?\)\s*\)/`，提高舊 DB 也能成功跑遷移的機率。

- **server/storage.ts**  
  - 引入 `CONTACT_STATUS_ALLOWED`。  
  - `updateContactStatus(id, status)`：若 `status` 不在 `CONTACT_STATUS_ALLOWED` 則 `throw new Error(...)`，並列出允許值，不再直接寫入非法值。

- **server/routes.ts**  
  - **POST /api/contacts/:id/assign**：整段指派邏輯包在 `try/catch`；catch 到錯誤時，若訊息為 `CHECK constraint failed` / `constraint failed` / `SQLITE_CONSTRAINT` 則回傳 500 與「指派時更新狀態失敗，請重新整理頁面後再試。若持續發生請聯絡管理員。」，其餘錯誤回傳 500 與簡短訊息（過長則改為「指派時發生錯誤，請稍後再試。」）。  
  - **POST /api/contacts/:id/reassign**：同樣用 try/catch，constraint 時回傳 500 與「改派時更新狀態失敗…」。

- **client/src/pages/chat.tsx**  
  - **handleAssignToAgentWith**：catch 時設定 `setShowAssignDialog(false)`、`setAssignAgentId(null)`；錯誤訊息優先使用 API 回傳的 `message`，若 raw 訊息含 CHECK/constraint 關鍵字則改為「指派時更新狀態失敗，請重新整理頁面後再試。」；`finally` 維持 `setAssigning(false)`。  
  - **handleAssign**（自動分配）：catch 時 `setShowAssignDialog(false)`；503 與 constraint 時顯示對應友善訊息。  
  - **handleReassignSubmit**：catch 時解析 API body、constraint 時顯示「改派時更新狀態失敗…」並 `setShowReassignDialog(false)`；`finally` 維持 `setReassigning(false)`。

---

## 4. 原本哪個 status 不合法

- 寫入的值是 **`assigned`**。  
- 指派／改派流程會呼叫 `storage.updateContactStatus(contactId, "assigned")`，若 DB 的 CHECK 裡沒有 `assigned`，就會觸發 `CHECK constraint failed: status IN (...)`。

---

## 5. 現在改成什麼正確邏輯

- **資料庫**：啟動時跑 `ensureContactStatusIncludesAssigned()`，若目前 `contacts` 的 CHECK 尚未包含 `assigned` / `waiting_customer` 等，就用遷移重建表並套用完整 `CASE_FLOW_STATUSES`（含 `assigned`, `waiting_customer`, `new_case`, `pending_info`, `pending_order_id`, `resolved_observe`, `reopened` 等），之後寫入 `assigned` 即合法。  
- **後端**：  
  - 只寫入 `CONTACT_STATUS_ALLOWED` 裡的值；`updateContactStatus` 在寫入前檢查，非法就拋錯。  
  - 指派／改派 API 捕獲 DB 錯誤，constraint 時回傳 500 與固定友善文案，不再把資料庫錯誤原文丟給前端。  
- **前端**：  
  - 不論成功或失敗，loading 一定在 `finally` 結束。  
  - 失敗時一律關閉對應 modal，並顯示後端回傳訊息或 constraint 專用文案，不再卡住或只顯示 DB 錯誤。

---

## 6. 怎麼測試

1. **重啟後端**（讓 DB 遷移與新 API 邏輯生效）。  
2. 用**主管帳號**登入，進入即時客服，選一筆「待分配」對話。  
3. 點「指派」→ 選一位客服 → 點「指派給選定客服」。  
4. 再測：同一筆或另一筆，點「指派」→ 直接點「自動分配（依負載與時段）」。  
5. 再測：已指派的對話點「改派」→ 選另一位客服 → 確認改派。

---

## 7. 驗收步驟

- 指派客服（手動指定）成功：不報錯、modal 關閉、右側負責人與列表更新為該客服、列表可刷新。  
- 指派客服（自動分配）成功：同上。  
- 改派成功：不報錯、改派 modal 關閉、負責人與列表更新為新客服。  
- 指派失敗（例如 503 無可接案）：不卡住、modal 關閉、toast 顯示「目前無可接案客服…」或後端訊息、可再開 modal 或選其他對話。  
- 若仍出現 DB constraint（例如遷移未跑）：toast 顯示「指派時更新狀態失敗，請重新整理頁面後再試。」、modal 關閉、loading 結束；重新整理後重試應可成功（遷移會在啟動時跑）。

---

## 8. 驗收成功標準

- 成功時：不報錯、modal 正常關閉、畫面與列表狀態正確更新、資料刷新正確。  
- 失敗時：不卡住、loading 結束、modal 可關閉或已自動關閉、顯示人類可理解的錯誤訊息（無 DB 原文）。

---

## 9. 若再失敗，下一步怎麼查

1. **仍出現 CHECK constraint**  
   - 看後端啟動 log 是否有「補強 contacts.status CHECK」或「已支援案件流程狀態（含 assigned）」；若沒有，表示遷移沒跑，檢查 `db.ts` 裡 `ensureContactStatusIncludesAssigned` 的 regex 與目前 DB 的 `contacts` 表 schema（`SELECT sql FROM sqlite_master WHERE name='contacts'`）是否對得上。  
   - 手動確認該表 CHECK 是否包含 `assigned`，若不包含，可依 `CASE_FLOW_STATUSES` 手動改表或再跑一次遷移邏輯。

2. **錯誤訊息仍是資料庫原文**  
   - 確認前端是否已更新（強制重整 / 清快取），以及 catch 是否正確解析 `e.message` 與 response body 的 `message`。  
   - 確認後端 assign/reassign 的 catch 是否有回傳 `{ message: "…" }` 且 status 500。

3. **modal 仍卡住或 loading 不結束**  
   - 確認 `setAssigning(false)` / `setReassigning(false)` 在 `finally` 裡。  
   - 確認 catch 裡有 `setShowAssignDialog(false)` / `setShowReassignDialog(false)`。

---

## 10. 還有哪些地方會寫同一個 status 欄位（可能觸發同樣 constraint）

以下都會呼叫 `storage.updateContactStatus(..., status)`，寫入的值若不在 DB CHECK 內就會觸發同樣問題；目前寫入的都是 `CONTACT_STATUS_ALLOWED` 內的值，且已用該常數在 `updateContactStatus` 擋掉非法值：

| 位置 | 寫入的 status |
|------|----------------|
| `server/assignment.ts` | `assigned`（assignCase、assignCaseManual、reassignCase、tryReassignOverdue）、`awaiting_human`（unassignCase） |
| `server/routes.ts` | `awaiting_human`（transfer-human、AI 流程）、`ai_handling`（restore-ai、AI 流程）、`high_risk`（AI 流程）、手動 PUT status 時由前端傳入（已用 validStatuses 白名單） |

`routes.ts` 的 `validStatuses` 陣列已包含上述與 `assigned` 等，若未來新增 status，請同步：  
- `shared/schema.ts` 的 `ContactStatus` 與 `CONTACT_STATUS_ALLOWED`  
- `server/db.ts` 的 `CASE_FLOW_STATUSES`（並確保遷移有跑）  
- `server/routes.ts` 的 `validStatuses`（若為 API 可設定的狀態）
