**ensurePerformanceIndexes() 完整程式碼**（節錄自 server/db.ts 第 159–191 行）：

```ts
/** 常用查詢欄位索引，避免資料量成長後掃全表；MAX(id) GROUP BY 等 Stats 依賴複合索引避免 Full Table Scan */
function ensurePerformanceIndexes() {
  const run = (sql: string) => {
    try {
      db.exec(sql);
    } catch (e) {
      console.warn("[DB] Index creation skipped:", (e as Error).message, sql.slice(0, 60) + "...");
    }
  };

  // 極速複合索引（解決 getContacts / Stats 中 MAX(id) GROUP BY 等 Full Table Scan）
  run(`CREATE INDEX IF NOT EXISTS idx_messages_contact_id_id ON messages(contact_id, id DESC);`);
  run(`CREATE INDEX IF NOT EXISTS idx_messages_sender_created ON messages(sender_type, created_at DESC);`);
  run(`CREATE INDEX IF NOT EXISTS idx_contacts_brand_status ON contacts(brand_id, status);`);
  run(`CREATE INDEX IF NOT EXISTS idx_contacts_assigned_agent ON contacts(assigned_agent_id);`);
  run(`CREATE INDEX IF NOT EXISTS idx_contacts_last_message_at ON contacts(last_message_at DESC);`);
  /** WHERE brand_id + status 且 ORDER BY last_message_at 時可覆蓋，避免 filesort */
  run(`CREATE INDEX IF NOT EXISTS idx_contacts_brand_status_last_message_at ON contacts(brand_id, status, last_message_at DESC);`);

  run(`CREATE INDEX IF NOT EXISTS idx_messages_contact_id ON messages(contact_id);`);
  run(`CREATE INDEX IF NOT EXISTS idx_messages_contact_created ON messages(contact_id, created_at DESC);`);
  run(`CREATE INDEX IF NOT EXISTS idx_messages_contact_id_desc ON messages(contact_id, id DESC);`);
  run(`CREATE INDEX IF NOT EXISTS idx_contacts_brand_id ON contacts(brand_id);`);
  run(`CREATE INDEX IF NOT EXISTS idx_contacts_assigned_agent_id ON contacts(assigned_agent_id);`);
  run(`CREATE INDEX IF NOT EXISTS idx_contacts_platform_user ON contacts(platform, platform_user_id);`);
  run(`CREATE INDEX IF NOT EXISTS idx_contacts_status ON contacts(status);`);
  run(`CREATE INDEX IF NOT EXISTS idx_ai_logs_contact_id ON ai_logs(contact_id);`);
  run(`CREATE INDEX IF NOT EXISTS idx_ai_logs_created_at ON ai_logs(created_at DESC);`);

  // 反正規化：contacts 快取最後一則訊息內容與發送者類型，消除 getContacts 的 JOIN/MAX(id) 瓶頸
  run("ALTER TABLE contacts ADD COLUMN last_message_content TEXT;");
  run("ALTER TABLE contacts ADD COLUMN last_message_sender_type TEXT;");
}
```

### B3. storage.ts getContacts 完整修改內容

**現行實作**：單表 contacts + LEFT JOIN brands / channels / users，**不再查 messages 表**；最後一則訊息來自 contacts 反正規化欄位 `last_message_content`、`last_message_sender_type`。

```ts
  const params: any[] = [];
  const conditions: string[] = [];

  query += " ORDER BY c.is_pinned DESC, (CASE WHEN c.case_priority IS NULL THEN 999 ELSE c.case_priority END) ASC, c.last_message_at DESC";

  const withPreview = contacts.map((c) => {
    const rawType = (c as any).last_message_sender_type;
    const senderType = ...;
    return { ...c, last_message: (c as any).last_message_content ?? "", last_message_sender_type: senderType ?? undefined };
  });
  // agentId 時再查 getAgentContactFlags（已分批 500）
  // ...
  return withPreview;

```

**修改前（已移除）**：原先有一段依 contactIds 做 `IN (?)` + `SELECT contact_id, MAX(id) ... FROM messages ... GROUP BY contact_id` 再 JOIN messages 取 content/sender_type；此子查詢在 contactIds 多時會造成大量變數與 Full Table Scan / USE TEMP B-TREE FOR GROUP BY。

### B4. EXPLAIN QUERY PLAN Before（舊版 messages 子查詢，僅對照用）

舊版子查詢型態（已不再使用）：

```sql
SELECT m.contact_id, m.content, m.sender_type
FROM messages m
INNER JOIN (
  SELECT contact_id, MAX(id) AS max_id
  FROM messages
  WHERE contact_id IN (1,2,3,4,5)
  GROUP BY contact_id
) t ON m.contact_id = t.contact_id AND m.id = t.max_id
```

執行 `EXPLAIN QUERY PLAN` 對照結果（小 IN 時）：

```
CO-ROUTINE t
SEARCH messages USING COVERING INDEX idx_messages_contact_id (contact_id=?)
SCAN t
SEARCH m USING INTEGER PRIMARY KEY (rowid=?)
```

當 contactIds 很多（例如數百）時，IN 變數過多會觸發 SQLite 限制，且易出現 **USE TEMP B-TREE FOR GROUP BY**；getContacts 主路徑已完全移除此子查詢。

### B5. EXPLAIN QUERY PLAN After（目前 getContacts 主查詢）

**主查詢（無 WHERE）**：

```
SCAN c
SEARCH b USING INTEGER PRIMARY KEY (rowid=?) LEFT-JOIN
SEARCH ch USING INTEGER PRIMARY KEY (rowid=?) LEFT-JOIN
SEARCH u USING INTEGER PRIMARY KEY (rowid=?) LEFT-JOIN
USE TEMP B-TREE FOR ORDER BY
```

**主查詢（WHERE brand_id = 1）**：

```
SEARCH c USING INDEX idx_contacts_brand_id (brand_id=?)
SEARCH b USING INTEGER PRIMARY KEY (rowid=?) LEFT-JOIN
SEARCH ch USING INTEGER PRIMARY KEY (rowid=?) LEFT-JOIN
SEARCH u USING INTEGER PRIMARY KEY (rowid=?) LEFT-JOIN
USE TEMP B-TREE FOR ORDER BY
```

結論：**已無 SCAN messages**；有 brand_id 時為 **SEARCH contacts USING INDEX**；ORDER BY 仍用 TEMP B-TREE（因含 is_pinned、case_priority、last_message_at 複合排序）。

### B6. API / Query 耗時 Before / After

- **Before**：GET /api/contacts 曾出現約 **495ms** 延遲（瓶頸在 messages 子查詢 + GROUP BY）。
- **After**：執行 `npx tsx server/scripts/explain-get-contacts.ts` 量測 **storage.getContacts** 10 次：
  - **平均**: 1 ms  
  - **p95**: 2 ms  
  - **最慢**: 2 ms  
  - 10 次依序(ms): 0, 0, 0, 0, 0, 1, 1, 1, 1, 2  

證明查詢改為單表 + 反正規化後延遲明顯下降。

### B7. 為什麼這版查詢會吃到索引

- getContacts **不再查 messages**，只查 contacts + LEFT JOIN brands/channels/users；因此不會出現 **SCAN messages**。
- 有 `WHERE c.brand_id = ?` 時使用 **idx_contacts_brand_id**（或可選 idx_contacts_brand_status_last_message_at）。
- ORDER BY 使用 last_message_at 時，**idx_contacts_last_message_at** 可供排序使用；有 brand_id + status 時 **idx_contacts_brand_status_last_message_at** 可同時涵蓋篩選與排序。

### B8. 是否新增額外複合索引（brand_id, status, last_message_at DESC）

**是，已新增**：

```ts
run(`CREATE INDEX IF NOT EXISTS idx_contacts_brand_status_last_message_at ON contacts(brand_id, status, last_message_at DESC);`);
```

**原因**：GET /api/contacts 常帶 `brand_id`（與選用 `status`）且依 `last_message_at DESC` 排序；此複合索引可讓「WHERE brand_id = ? AND status = ? ORDER BY last_message_at DESC」盡量以索引完成，減少 TEMP B-TREE 用於 ORDER BY。

### C7. 如何實現 debounce / dedupe / per-contact 串行

- **debounce**：目前為 **記憶體版** `messageDebounceBuffers`（Map<contactId, { texts, timer }>）在 `server/routes.ts` 的 `debounceTextMessage`。多實例部署會失效，規格要求需改為 Redis 或 BullMQ delayed job + 聚合，**尚未改為分散式**。
- **dedupe**：以 `eventId`（LINE: webhookEventId / timestamp；FB: fb_${timestamp}_${senderId}_${mid}）呼叫 `storage.isEventProcessed(eventId)` / `storage.markEventProcessed(eventId)`，避免同一事件重複處理。
- **per-contact 串行**：目前 **未強制**。addAiReplyJob 的 jobId 為 `contactId + "_" + Date.now() + "_" + random`，同一 contact 可有多個 job 並行。規格要求需以 Redis lock 或 BullMQ 單一 job 佇列（例如 per-contact jobId 或 Redis 鎖）保證同一 contact 串行，**尚未實作**。

### C8. 如何避免 retry 造成重複回覆

- 去重以 **inbound event id** 為準（見 C7），同一 event 只會產生一筆 job；retry 時 job 為同一筆，若 processor 內先寫入「已回覆」狀態再發送，可做冪等。
- 目前 **未** 在 DB/Redis 記錄「此 message_id / event_id 已由 AI 回覆」，retry 時有可能重複發送。建議：在成功發送後寫入 processed_events 或專表（例如 `ai_reply_sent(event_id)`)，processor 開頭檢查已處理則 skip，以達冪等。

### C9. 如何確保 route enqueue 後立即回 200

- **LINE**：handleLineWebhook 在驗簽與基本檢查後立即 `res.status(200).json({ success: true })`，再以 `(async () => { for (event) { ... debounceTextMessage(..., () => addAiReplyJob(...)); } })()` 處理，**不等待** addAiReplyJob 或 autoReplyWithAI。
- **FB**：同上，先 `res.status(200).send("EVENT_RECEIVED")`，再 `(async () => { ... })()` 內呼叫 addAiReplyJob，**不等待**。

## 附錄：啟動證據（ensurePerformanceIndexes 有執行）

啟動 server 時會載入 `server/storage.ts`，其頂層執行 `initDatabase()`，而 `initDatabase()` 內會呼叫 `ensurePerformanceIndexes()`。若索引已存在，`run()` 內 try-catch 會略過重複建立並在 console 出現類似：

```
[DB] Index creation skipped: ... (若該索引已存在)
```

首次建立時則無錯誤。可於啟動 log 中確認無因索引建立失敗而拋錯，即表示已執行。

執行 EXPLAIN 與耗時腳本：

```bash
npx tsx server/scripts/explain-get-contacts.ts
```

可再次驗證查詢計畫與 getContacts 耗時（平均約 1ms、p95 2ms）。
