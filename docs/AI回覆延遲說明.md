# AI 回覆延遲說明：為何會等三分鐘或更久？

## 一句話

客人發訊後到收到 AI 回覆，會經過：**防抖 → 佇列等待 → 回覆流程（含可能的多輪 LLM + 訂單 API）**。其中 **訂單查詢** 與 **佇列／LLM 耗時** 最容易造成「問了三分鐘才得到回覆」。

---

## 流程概覽（從客人發訊到回覆）

```
客人發訊
  → [1] 防抖 (Debounce) 約 1.2 秒
  → [2] 送入 Redis 佇列（delay 1.2s 後才執行）
  → [3] Worker 取 job，呼叫 /internal/run-ai-reply
  → [4] 回覆流程：狀態解析、回覆計畫、必要時轉人工
  → [5] 若走 AI：組 system prompt、呼叫 OpenAI（主回覆）
  → [6] 若 AI 呼叫「查單」工具：對外訂單 API 逐頁請求（可能 20s～70s+）
  → [7] 若有 tool call：再一輪 LLM 產生最終回覆
  → [8] 寫入 DB、推播、送 LINE/FB
```

所以「問了三分鐘才回」通常是：**[2] 等佇列 + [5]～[7] 多輪 LLM + [6] 查單 API 很慢** 加總的結果。

---

## 延遲主要來源

| 階段 | 說明 | 典型耗時 |
|------|------|----------|
| **防抖** | 同一位客人短時間多則合併，避免重複觸發 | 固定約 **1.2 秒** |
| **佇列等待** | 同時只有 5 個 job 在跑，其餘在「等待」 | 視負載，可能 **0～數分鐘** |
| **主流程** | 狀態解析、回覆計畫、組 prompt、第一次 LLM | **約 10～30 秒**（依模型與 token 數） |
| **訂單查單** | 依商品+電話／日期+聯絡人呼叫一頁商店等 API，**逐頁**請求，最多 25～40 頁 | **20～70+ 秒**（每頁 1～3 秒 × 頁數） |
| **第二輪 LLM** | 有 tool call 時，把工具結果再送給模型產出最終回覆 | **約 10～30 秒** |

加總起來，**查單情境** 出現 **2～3 分鐘甚至更久** 是可能的。

---

## 如何確認延遲出在哪裡（Log）

已加上分段耗時 log，依時間順序會看到：

| Log 關鍵字 | 意義 |
|------------|------|
| **`run-ai-reply start contactId=...`** | Worker 開始處理這則訊息（此時已過防抖 + 佇列等待） |
| **`after_plan_ms`** | 狀態解析、回覆計畫（buildReplyPlan）完成，尚未呼叫 LLM |
| **`after_first_llm_ms`** | 第一次 OpenAI 呼叫完成（若沒 tool call，接下來就會送最終回覆） |
| **`tool <名稱> ms`** | 某個 tool（如查單）執行耗時，每個 tool 都會有一行 |
| **`reply_sent_total_ms`** | 最終回覆已寫入並送出，總耗時；`tools=` 列出有呼叫的 tool |
| **`run-ai-reply done ... in Xms`** | 整段 run-ai-reply（含 DB、推播）總共 X 毫秒 |

解讀方式：

- **after_plan_ms 很小、after_first_llm_ms 很大**：第一次 LLM 本身慢（模型或 token 多）。
- **tool lookup_order_... ms 很大**：查單 API 是瓶頸（逐頁請求、頁數多或網路慢）。
- **reply_sent_total_ms 很大但 tools=none**：沒查單也慢，多半是 LLM 或佇列等待。
- **run-ai-reply done in 180000ms**（3 分鐘）：從 worker 接到 job 到回 200 的總時間；對照上面分段可知是「等 LLM」還是「等 tool」。

## 可調整的優化方向

1. **縮短防抖**  
   - 檔案：`server/routes.ts`（約 248 行）、`server/queue/ai-reply.queue.ts`（約 22 行）  
   - 常數：`DEBOUNCE_MS = 1200`  
   - 可改為 `600`～`800`，會稍快開始處理，但連發多則時合併效果會變短。

2. **佇列並行數**  
   - 檔案：`server/queue/ai-reply.queue.ts`  
   - `AI_REPLY_CONCURRENCY = 5`：同時處理 5 個 job。  
   - 若機器與 API 負載允許，可適度提高，減少「等佇列」時間；需注意對外 API（訂單、OpenAI）限流。

3. **訂單 API 耗時（影響最大）**  
   - 查單時會對一頁商店（或 Shopline）**逐頁**請求，每頁 200 筆、最多 25～40 頁。  
   - 若訂單量大，容易出現 20～70+ 秒。  
   - 可行方向（需改程式）：  
     - 限制前幾頁就先回一版（例如「先顯示部分，其餘查詢中」）、  
     - 或對「依日期+聯絡人」等情境做短期快取，減少重複打 API。

4. **模型與 token**  
   - 使用較快模型或縮短 system prompt／歷史則數，可減少 [5][7] 的 LLM 時間。

---

## 總結

- **「問了三分鐘才得到回覆」** 多半是：**防抖 + 佇列 + 一次或多次 LLM + 查單 API 逐頁請求** 加總。
- 用 **`[AI Latency]`** 的 log 可看出：是 **整段慢** 還是 **查單 tool 特別慢**。
- 優先可做：**縮短 DEBOUNCE_MS**、**觀察並適度提高 concurrency**、**優化或限頁／快取訂單查詢**。
