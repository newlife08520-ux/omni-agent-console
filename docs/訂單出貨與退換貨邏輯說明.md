# 訂單出貨進度查詢與退換貨邏輯（完整說明）

## 一、詢問訂單／出貨進度邏輯

### 1. 觸發條件（誰會走「訂單查詢」）

- **對話狀態（ConversationState）**：`conversation-state-resolver` 會依客戶訊息判斷 `primary_intent`。
- **訂單查詢意圖**：當客戶訊息或近期訊息出現「訂單、查單、出貨、物流、還沒到、單號、編號」，或**單則訊息為 5 碼以上英文數字**（如 `MRQ55276`）時，`primary_intent` = **`order_lookup`**。
- **Reply Plan**：若 `primary_intent === "order_lookup"` 且**不是**退換貨意圖（非 refund_or_return / exchange_request / cancellation_request），則 `plan.mode = "order_lookup"`，本輪以訂單查詢為主。

### 2. 退換貨時「不先查單」

- 若客戶同時提到退貨／退款／換貨／取消，且為**退換貨首輪**（`return_stage` 為 0 或未設），則 `plan.mode = "return_form_first"`，**不會**走訂單查詢主流程。
- 系統會在本輪 system prompt 加上：「本輪為退換貨承接，不可先走訂單查詢為主流程；先承接需求並引導表單。」
- 因此「我想退貨，MRQ55276」會先得到退換貨承接＋表單，不會先查單。

### 3. 訂單查詢工具（三種）

AI 可呼叫以下工具，由 `order-service` 統一查一頁商店 + SHOPLINE：

| 工具名稱 | 用途 | 必要參數 | 說明 |
|----------|------|----------|------|
| **lookup_order_by_id** | 用訂單編號查 | `order_id` | 客戶提供訂單編號（如 KBT58265、MRQ00001）時使用。系統會自動轉大寫。先查一頁商店，再查 SHOPLINE；必要時會跨品牌查其他品牌一頁商店。 |
| **lookup_order_by_product_and_phone** | 商品＋手機查 | `phone`，以及 `product_name` 或 `product_index` | 客戶沒有單號、但有「買什麼商品＋下單手機」時使用。依商品名稱對應銷售頁（page_id），再用 page_id + 手機查一頁商店／SHOPLINE。**禁止僅用手機做全域搜尋**，須先有商品。 |
| **lookup_order_by_date_and_contact** | 日期＋聯絡人查 | `contact`、`begin_date`、`end_date`、`page_id` | 客戶提供下單日期區間與 Email／手機／姓名，且能對應到商品（有 page_id）時使用。日期範圍有限制（例如 31 天）。 |

### 4. 編號類型過濾（避免誤查）

- 呼叫 **lookup_order_by_id** 前，會用 `classifyOrderNumber` 判斷輸入是訂單編號、金流序號還是物流單號：
  - **order_id**：如 KBT、ORD、MRQ、MRH、DEN 開頭或 2–4 碼英文+數字，視為訂單編號 → 實際查單。
  - **payment_id**：如 T 開頭 10 碼、或出現「交易序號／金流」等 → 不查單，回覆「這組比較像付款／金流資訊，不是訂單編號」，並引導提供訂單編號或收件人姓名＋手機後三碼。
  - **logistics_id**：如 10–14 位數字（非 09 開頭）或宅配／物流關鍵字 → 不查單，回覆「這組比較像物流單號，不是訂單編號」，並引導提供訂單編號等。

### 5. 查詢結果處理

- **有查到**：回傳訂單狀態（一頁商店／SHOPLINE 的 status 會轉成統一狀態文案）、金額、商品、物流單號、出貨日等；AI 用自然語氣回覆客戶出貨／物流進度。
- **查無此筆**：
  - **lookup_order_by_id**：回傳 `found: false`，訊息為「所有平台（一頁商店 + SHOPLINE）皆查無訂單編號 XXX 的紀錄。請告知客戶目前查不到這筆資料，並詢問是否需要轉接專人客服。」
  - **lookup_order_by_product_and_phone / date**：類似「查無此手機號碼的訂單」或「指定日期範圍內均查無相符紀錄」。
- **不因查無就自動轉人工**：System prompt 明確寫「訂單查詢工具回傳 found=false 或空陣列時，用你的語氣告知客戶查不到資料，主動詢問是否有其他資訊可協助查詢，不要自動轉人工。」
- **多次查單失敗**：同一輪若 **lookup_order 類工具** 回傳 `found: false` 累計 **≥ 2 次**，且尚未呼叫 transfer_to_human，則系統會**自動**將該聯絡人設為待人工（awaiting_human、needs_human=1、建立 case 通知），避免客戶一直查不到卻沒人接手。

### 6. 訂單來源與轉人工

- 「訂單來源（一頁商店、SHOPLINE）」**不是**轉人工的判斷依據；僅在符合五大轉人工條件或多次查單失敗時才轉。

## 二、退換貨邏輯

### 1. 意圖辨識

- **關鍵字**（`RETURN_REFUND_KEYWORDS` / conversation-state）：退貨、退款、換貨、退錢、退費、取消訂單、不要了、不想等、等太久等 → `primary_intent` 為 **refund_or_return**、**exchange_request** 或 **cancellation_request**。
- **明確堅持**：若出現「我就是要退、直接幫我退、我不要其他方案、我就是要退款、不要再跟我說別的方法、我不接受其他處理方式」→ 同時設 **needs_human = true**、**human_reason = return_stage_3_insist**，本輪走 **handoff**（轉人工）。

### 2. 流程優先級（Reply Plan）

- **先判斷再說話**：每則客戶訊息會先跑 `resolveConversationState` → `buildReplyPlan`。
- 退換貨相關優先級（在「訂單查詢」之前）依 **return_reason_type** 分流（Phase 2 唯一版本）：
  1. 明確要真人、法務/投訴、金流/訂單爭議、明確堅持退款退貨 → **handoff**。
  2. **wait_too_long**（等太久、不想等）→ **aftersales_comfort_first**（先安撫＋查單，不先表單）。
  3. **product_issue**（瑕疵、損壞、錯貨、缺件）→ **return_form_first**（正式售後，先表單）。
  4. **insist**（我就是要退、直接幫我退）→ 不再挽回，**return_form_first** 或 handoff；return_stage === 1 → **return_stage_1**。
  5. 其後才是 **order_lookup**、answer_directly 等。  
  詳見 `docs/Phase2-mode與流程單一化.md`。

### 3. 退換貨首輪：return_form_first（固定文案）

- **條件**：`primary_intent` 為 refund_or_return / exchange_request / cancellation_request，且 needs_human 未觸發，且 `return_stage == null || return_stage === 0`。
- **行為**：**不呼叫 LLM**，直接回覆固定文案：
  - 「了解，這邊先幫您處理🙏 若您要申請退換貨，麻煩先幫我填寫退換貨表單，填完後我們的專人會接續協助您確認後續流程。」
  - 若有設定品牌 `return_form_url`，會附上「表單連結：{url}」。
  - 並補「若您是因為等太久、商品異常，或其他原因，也可以在表單裡一起備註，這樣我們處理會更快。」
- **寫入狀態**：`return_stage = 1`、`resolution_status = "awaiting_customer"`、`waiting_for_customer = "return_form_submit"`。

### 4. F2 規則（禁止第一句區分官方／其他平台）

- **適用**：退換貨相關回覆（return_form_first、return_stage_1、handoff 時仍須遵守）。
- **禁止出現**：官方通路、其他平台、若是其他平台購買、向該平台客服、非官方 等讓客戶覺得被推到別家的說法。
- **允許例外**：僅在**客戶已提供訂單編號或商品+手機**，且**已用工具查詢**，結果**確實查無此筆**時，才可禮貌說明可能為其他通路訂單並建議轉真人協助；且**不得在客戶尚未提供任何訂單資訊前就提其他平台**。
- **實作**：`plan.mode` 為 return_form_first / return_stage_1 / handoff 時，`plan.must_not_include` 會帶入 F2 禁止句，並注入 system prompt「本輪硬規則 F2」。

### 5. Prompt 內退換貨三階段（給 LLM 的規則）

當本輪**不是** return_form_first（即已給過表單或非首輪），改由 LLM 依下列規則回覆：

- **第一段**：先理解原因，不直接轉人工。客戶提到退貨、退款、取消、不想等、缺貨、久候時，先安撫、理解、收一次必要資訊。
- **第二段**：提供 AI 可處理的下一步（規則、可能做法、引導訂單編號／商品名稱／通路、表單時機等）；表單可在合適時機提供，**不可一開始就丟表單**。
- **第三段**：只有在明確要求真人、明確堅持退款退貨、AI 已處理一輪仍無法推進、問題屬人工權限、或情緒/法務/投訴風險過高時，才呼叫 **transfer_to_human**。

### 6. 轉人工條件（與退換貨相關）

- 退換貨情境下，**只有**以下情況可直接轉人工：
  - 顧客明確要求真人（explicit_human_request）。
  - 投訴／法務／公開風險（legal_or_reputation_threat）。
  - 金流異常、收到錯貨、補償、例外退款等（payment_or_order_risk / policy_exception）。
  - AI 已處理一輪仍無法解決（repeat_unresolved）。
  - **退換貨明確堅持**（return_stage_3_insist）：例如「我就是要退、直接幫我退」。
- 禁止：一看到退貨就答應退款、一看到退款就轉人工、還沒理解問題就直接給表單、因單一字詞（退貨、退款、客服、缺貨、取消）直接判定真人。

### 7. 表單連結來源

- 品牌設定 **return_form_url**（後台「退換貨表單連結」）；若未設則使用預設 `https://www.lovethelife.shop/returns`。
- 該 URL 會出現在 handoff 區塊的「售後表單」說明中，以及 **return_form_first** 固定文案裡。

## 三、流程簡圖（摘要）

```
客戶訊息
    ↓
resolveConversationState（primary_intent, needs_human, return_stage...）
    ↓
buildReplyPlan（returnFormUrl, isReturnFirstRound）
    ↓
┌─ plan.mode === "return_form_first" → 固定退換貨＋表單文案，寫 return_stage=1，return
├─ plan.mode === "handoff" → 加 handoff 提示，進 LLM，須呼叫 transfer_to_human
├─ plan.mode === "order_lookup" 且非退換貨首輪 → 可查單；若 shouldNotLeadWithOrderLookup 則加「不可先查單」
└─ 其餘 → 進 LLM，依 prompt 與工具回覆
    ↓
訂單查詢：lookup_order_by_id / by_product_and_phone / by_date_and_contact
    → found ? 回覆狀態與出貨進度 : 回覆查無、詢問其他資訊；≥2 次失敗則自動轉人工
退換貨（非 return_form_first）：依三階段與 F2，必要時 transfer_to_human
```
