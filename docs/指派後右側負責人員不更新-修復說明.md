# 指派成功後右側「負責人員」不更新 - 修復說明

## 1. 真正 bug 原因

- **右側「負責人員」** 的資料來源是 **assignmentData**，來自 **useQuery**，queryKey 為 `["/api/contacts", selectedId, "assignment"]`，即 **GET /api/contacts/:id/assignment**。
- **手動指派** 時前端呼叫的是 **POST /api/contacts/:id/assign**，後端成功後只回傳 `{ success: true, assigned_agent_id: number }`，**沒有回傳** `assigned_agent_name`、`assigned_agent_avatar_url` 等右側 UI 需要的欄位。
- 前端原先只做 **invalidateQueries + refetchQueries**。refetch 會再打一次 GET assignment，但：
  - 若 refetch 非同步、尚未回傳就已 render，右側仍會是舊資料；
  - 或 query 的 queryFn 與 cache 行為導致沒有馬上用新資料 re-render。
- 因此**根本原因**是：**指派成功後，右側所依賴的 assignment 資料沒有「立刻」被更新**。只依賴 refetch 不穩定，且 POST 回應裡本來就沒有姓名／頭像，無法在第一時間寫入 UI。

---

## 2. 改了哪些檔案

| 檔案 | 修改內容 |
|------|----------|
| **server/routes.ts** | POST assign、POST unassign、POST reassign 成功時，改為回傳與 GET assignment **同結構**的完整資料（assigned_to_user_id、assigned_agent_name、assigned_agent_avatar_url、assigned_at、assignment_method 等），供前端直接寫入 cache。 |
| **client/src/pages/chat.tsx** | 手動指派、自動分配、改派、移回待分配 四個流程，改為：**讀取 API 的 response body**，若 `data.success` 為 true，則用 **queryClient.setQueryData** 把回傳的 assignment 資料寫入 `["/api/contacts", contactId, "assignment"]`。右側「負責人員」綁定同一份 assignment 資料，因此會立即顯示姓名與照片，不需等 refetch。 |

---

## 3. 每個檔案改了什麼（摘要）

### server/routes.ts

- **POST /api/contacts/:id/assign** 成功後：  
  重新讀取 `storage.getContact(contactId)` 與指派對象 `storage.getUserById(agentId)`，回傳 `success`、`assigned_agent_id`、`assigned_to_user_id`、`assigned_at`、`assignment_status`、`assignment_method`、**assigned_agent_name**、**assigned_agent_avatar_url**、以及 last_human_reply_at、reassign_count、needs_assignment、response_sla_deadline_at 等，與 GET assignment 一致。
- **POST /api/contacts/:id/unassign** 成功後：  
  回傳同一結構，但 `assigned_to_user_id`、`assigned_agent_name`、`assigned_agent_avatar_url`、`assigned_at` 等為 null，表示待分配。
- **POST /api/contacts/:id/reassign** 成功後：  
  重新讀取 contact 與新客服 user，回傳完整 assignment 結構（含新客服的 name、avatar_url），assignment_method 為 "reassign"。

### client/src/pages/chat.tsx

- **handleAssignToAgentWith**（手動指派）：  
  `const res = await apiRequest(...)` 後 `const data = await res.json()`；若 `data?.success`，則 `queryClient.setQueryData(["/api/contacts", contactId, "assignment"], (prev) => ({ ...prev, ...payload, assigned_to_user_id: data.assigned_to_user_id ?? data.assigned_agent_id }))`，再 invalidate 列表、toast、關閉 modal。
- **handleAssign**（自動分配）：  
  同上，讀取 res.json()，成功則 setQueryData 寫入該 contactId 的 assignment，再 invalidate 與關閉 modal。
- **handleReassignSubmit**（改派）：  
  讀取 res.json()，成功則 setQueryData 寫入 assignment（含新客服姓名、頭像）。
- **handleUnassign**（移回待分配）：  
  讀取 res.json()，成功則 setQueryData 寫入 assignment（assigned_to_user_id 等為 null），右側會立即顯示「待分配」。

## 4. 右側「負責人員」原本是讀哪份資料

- **唯一來源**：**assignmentData**，來自 useQuery，queryKey 為 `["/api/contacts", selectedId, "assignment"]`，即 **GET /api/contacts/:id/assignment** 的回傳。
- 顯示邏輯：  
  - `assignmentData?.assigned_to_user_id` 有值 → 顯示頭像（assigned_agent_avatar_url）、姓名（assigned_agent_name）、指派方式。  
  - 無值 → 顯示「待分配」與「指派」按鈕。

---

## 5. 現在改成怎麼更新

- **後端**：POST assign / unassign / reassign 成功時，直接回傳與 GET assignment **相同結構**的完整資料（含 assigned_agent_name、assigned_agent_avatar_url）。
- **前端**：上述四個流程在收到 `data.success === true` 後，**立即** `queryClient.setQueryData(..., data)` 寫入對應 contactId 的 assignment cache，不再依賴 refetch 時機。  
- 右側「負責人員」仍讀同一份 assignmentData（同一 queryKey），因此會在同一次 render 就顯示最新姓名與照片（或待分配）。

1. **手動指派**：選一筆待分配 → 指派給某客服 → 確認右側「負責人員」**立即**顯示該客服姓名與照片（若有），且不重整頁面。
2. **自動分配**：選一筆待分配 → 自動分配 → 右側**立即**顯示被分配到的客服姓名與照片。

## 8. 如果還是不顯示，下一步該怎麼查

1. **Network**：  
   對 POST assign / reassign / unassign 看 response body 是否含 `assigned_agent_name`、`assigned_agent_avatar_url`。若沒有，表示後端未回傳完整，檢查 routes 是否已部署、是否有舊版 API。
2. **前端 setQueryData**：  
   在 setQueryData 後加 `console.log(["/api/contacts", contactId, "assignment"], payload)`，確認寫入的 payload 有 name、avatar；再在右側區塊加 `console.log(assignmentData)`，確認該 query 讀到的是同一 key 的資料。
3. **queryKey 是否一致**：  
   useQuery 的 queryKey 是 `["/api/contacts", selectedId ?? 0, "assignment"]`，setQueryData 用的必須是同一個 contactId（與 selectedId 一致），否則右側會讀到別筆或舊 cache。
4. **後端 GET assignment**：  
   若懷疑 GET /api/contacts/:id/assignment 回傳不完整，可直接打該 API 看回應是否含 assigned_agent_name、assigned_agent_avatar_url；POST 回傳結構已與 GET 對齊，理論上應一致。
