# 登入

## 1. 頁面用途
- **做什麼**：以帳號／密碼登入 Omni-Agent Console，建立 session 後進入 `/`。
- **給誰**：所有尚未通過 `GET /api/auth/check` 的使用者。
- **情境**：開啟站台、session 過期、手動登出後再次進入。

## 2. 進入方式
- **route**：無專用 `/login`；`AppContent` 在 `auth/check` 失敗、未認證或 `user` 異常時**全畫面**渲染 `LoginPage`（URL 仍可能為 `/` 等，**不以 path 區分**）。
- **從哪進**：開啟應用、session 過期、登出後重新載入。
- **權限**：無；需帳密正確。

## 3. 首屏必須看到的內容（3–5 區塊）
1. 標題「全通路 AI 客服中控台」與副標 `data-testid="text-login-title"`。
2. 帳號輸入 `input-username`。
3. 密碼輸入 `input-password`（可切換顯示／隱藏）。
4. 提交按鈕（登入）。
5. （若有）錯誤 toast — 非固定區塊但必須可感知。

## 4. 區塊排列順序
- 垂直置中單欄 `max-w-[360px]`：標題 → 表單（帳號 → 密碼+眼睛按鈕）→ 主按鈕。

## 5. 按鈕盤點
### A. 主按鈕（Primary）
- **登入提交**：完成認證；應在表單底部全寬或主視覺。
### B. 次按鈕
- **顯示/隱藏密碼**：`type="button"` 於密碼欄右側。
### C. 危險
- 本頁無典型 destructive；錯誤次數限制若後端有，**待確認**是否另顯示。
### D. 輔助
- 無側欄、無額外 utility。

## 6. 篩選 / 控制元件
- 無 tab／品牌；僅表單欄位。

## 7. 狀態 / badge / warning
- **Loading**：送出中按鈕 disabled 或 loading（程式內 `loading` state）。
- **錯誤**：toast「登入失敗」destructive。
- **成功**：toast「登入成功」後約 300ms 導向 `/`。

## 8. 可收合但不能消失
- 無收合區；標題與兩欄位不可刪。

## 9. 最不能被 AI 誤改
- 表單必須獨立於行銷 landing，不可改成「社群登入-only」而移除帳密。
- `credentials: "include"` 語意之 session 流程不可因 UI 改版被略過說明（營運文件層面）。

## 10. Stitch 用摘要（約 200 字）
登入頁為單欄置中表單：標題「全通路 AI 客服中控台」、帳號、密碼（可顯示密碼）、主按鈕登入。主 CTA 僅「登入」；次為密碼可見性切換。無側欄。錯誤與成功依 toast。視覺可優化背景與卡片圓角，但不可移除雙欄位與提交行為，不可改成需多步才能輸入帳密。桌面優先、寬度約 360px。
