## 四、生產環境必要環境變數（Railway）

若 **SESSION_SECRET** 或 **REDIS_URL** 未設，伺服器會直接 **exit(1)**，網站完全上不去。

| 變數 | 必填 (production) | 說明 |
|------|-------------------|------|
| `NODE_ENV` | 通常由 Railway 設為 `production` | 為 production 時會強制檢查下列兩項 |
| `SESSION_SECRET` | ✅ 必填 | 未設時：`FATAL: In production, SESSION_SECRET must be set. Refusing to start.` |
| `REDIS_URL` | ✅ 必填 | 未設時：`FATAL: REDIS_URL must be set in production. Refusing to start.` |
| `PORT` | 選填 | 預設 8080，Railway 多會注入 |
| `APP_DOMAIN` | 選填 | 用於日誌顯示 Webhook URL，不影響一般瀏覽 |

**請在 Railway 專案 → Variables 確認：**

- `SESSION_SECRET`：至少一組隨機長字串（例如 32 字元以上）
- `REDIS_URL`：Railway Redis 或外部 Redis 連線字串（例如 `redis://default:xxx@xxx.railway.app:port`）

## 四之二、AI 自動回覆（需另跑 Worker）

若 **LINE/FB 訊息有進中控台，但客戶收不到 AI 自動回覆**，多半是 **AI Reply Worker 沒有在跑**。

| 變數 / 設定 | 說明 |
|-------------|------|
| **Worker 進程** | 自動回覆由**獨立 Worker** 執行。本機需另開終端跑 `npm run dev:worker`；Railway 需**第二個 Service**，Start 指令為 `npm run start:worker`。 |
| `INTERNAL_API_SECRET` | 主站與 Worker **都要設**且**值相同**，Worker 呼叫 `/internal/run-ai-reply` 時會帶此 secret。 |
| `INTERNAL_API_URL`（Worker 用） | Worker 要打的主站網址。Railway 上若 Worker 與主站為不同 Service，填主站對外網址，例如 `https://你的主站.up.railway.app`。 |

詳見 **`docs/為什麼不會自動回覆.md`**。

## 五、若仍「上不去」的排查順序

1. **完全打不開 / 502 / 503**  
   - 看 Railway 的 **Deploy Log**：是否有 `FATAL: ... SESSION_SECRET ...` 或 `REDIS_URL ... Refusing to start`。  
   - 若有 → 補上對應環境變數後重新部署。

2. **能開登入頁，但登入後白屏 / 「主畫面載入錯誤」**  
   - 多為前端 TDZ（例如 `Cannot access 'x' before initialization`）。  
   - 目前程式已：切斷 chat 對 schema 的 runtime 依賴、ChatPage 改直接 import、ErrorBoundary 順序正確。  
   - 若仍發生：請在瀏覽器 Console 截圖完整錯誤與 stack（若已設 `minify: false`，變數名會較易對照）。

3. **登入按下去沒反應 / 404**  
   - 確認網址是 **正式站網址**（例如 `https://richbear-omnicare-hub.up.railway.app`），不是 `localhost:4173`（preview 無後端）。  
   - 開發時用 `npm run dev` 或 `npm run build` + `npm run start`，才能同時有 API 與前端。

4. **靜態檔 404（/assets/xxx.js 找不到）**  
   - 確認部署流程有執行 `npm run build`，且 `dist/public` 有被一併部署。  
   - Railway 若用 `npm run start` 啟動，需確保 start 前已 build（或部署時已產生 dist）。
