# 決策規則後續可維護方案（設計）

本文件記錄「強售後詞」與「direct_hide + 白名單」的下一步落地方案，供長期維護與下一輪實作參考。

---

## 一、STRONG_AFTER_SALE_KEYWORDS 抽成可維護設定

### 現狀
- 強售後詞目前為程式常數 `STRONG_AFTER_SALE_KEYWORDS`（server/meta-comment-risk-rules.ts）。
- 用於：A-2（hide_and_route 僅在「有強售後詞」時才壓過白名單）、A-3（含強售後詞不進 gray_area）。
- 短期可接受，長期營運需自行補詞，不應每次改程式。

### 建議落地方案

**方案 A：獨立設定表（推薦）**
- 新增 DB 表 `meta_strong_after_sale_keywords`（或 `meta_decision_config` 的 key-value）。
- 欄位建議：`id`, `keyword`（單詞）, `enabled`, `brand_id`（可選，空=全品牌）, `created_at`。
- 後台：在「留言風險與導流規則」同一區塊新增分頁或區塊「強售後詞管理」，可新增/刪除/啟用關鍵字。
- 決策引擎：啟動時或每次評估前讀取該表（可快取），`messageContainsStrongAfterSale(msg)` 改為查詢此表而非常數陣列。
- 優點：與現有規則分離，不影響五桶規則；營運可自助維護。

**方案 B：整合進現有規則桶**
- 不新增表，改為「視為一類特殊規則」：例如 rule_bucket = `strong_after_sale`（或沿用 whitelist 但加 tag）。
- 缺點：與現有五桶語意不同（強售後詞是「條件」不是「動作」），易混淆，不建議。

**方案 C：設定檔 / 環境**
- 以 JSON 檔或環境變數列出關鍵字，部署時更新。
- 優點：實作快。缺點：仍需改檔或重啟，營運無法自助。

**建議**：優先採 **方案 A**。實作步驟可為：
1. 新增 migration 與 CRUD API（GET/POST/DELETE 強售後詞）。
2. 規則管理頁加「強售後詞」子區塊，可新增/刪除/啟用。
3. `meta-comment-risk-rules.ts` 改為從 DB 讀取（含快取，避免每次請求都查）。
4. 若無任何關鍵字，可 fallback 回現有常數或空陣列，避免行為突變。

## 二、direct_hide + whitelist 不一律走白名單（技術債／下一輪）

### 現狀
- A-1：同句命中 direct_hide 與 whitelist 時，一律改走白名單，reason「命中競品詞但同時為正常詢問，避免誤殺」。
- 目的：避免「蝦皮有嗎？官網怎麼買」被誤隱藏。

### 風險
- 若句子同時帶白名單詞（如「價格」「官網」「怎麼買」）但語氣明顯是**負向亂入或攻擊**，一律白名單可能不恰當。
- 例：「官網價格寫假的吧」「怎麼買都是雷」等，可能不應完全白名單化。

### 建議下一步（列為技術債／下一輪）
- **輔助判斷**（擇一或組合）：
  - **購買意圖 / 問句型**：簡易 heuristics（如含「嗎」「呢」「請問」「想買」「哪裡買」且無強負評詞）才傾向白名單。
  - **強負評詞**：若同句出現「爛」「騙」「雷」「假」「垃圾」等，即使有白名單詞，仍可維持 direct_hide 或改走 gray_area，不一律白名單。
- **實作方式**：可在 `decideFinalRule` 的「direct_hide + whitelist 同時命中」分支，先做一層「是否像正常詢問」判斷（例如：無強負評詞 + 有問句特徵 → 走白名單；否則維持 direct_hide 或再細分）。
- 本輪不實作，僅在程式註解與本文件標註為**技術債**，下一輪再評估是否加入「強負評詞」常數或規則組，與「問句型」簡易判斷。
