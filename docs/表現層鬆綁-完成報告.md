# 表現層鬆綁完成報告：降低規則感、不與舊邏輯打架

## 一、哪些核心邏輯完全沒動

以下模組與流程**未做任何程式邏輯或分支改動**，僅在 prompt 用語上做「表現層」調整：

| 項目 | 狀態 |
|------|------|
| **conversation-state-resolver** | 未改。意圖判斷、human_request、order_lookup、off_topic、emotion 等邏輯與門檻不變。 |
| **reply-plan-builder** | 未改。mode 優先級、handoff / return_form_first / order_lookup / off_topic_guard 觸發條件不變。 |
| **handoff 觸發主邏輯** | 未改。何時轉人工、午休/下班明講、固定 handoff 句由程式送出等邏輯不變。 |
| **查單規則** | 未改。仍只允許「訂單編號」或「產品名稱＋手機」；工具層與路由邏輯不變。 |
| **already_provided 三層搜尋** | 未改。搜尋流程與「找不到就轉人工」行為不變。 |
| **content-guard 核心** | 未改。platform guard、official channel guard、no-promo、category mismatch、output guard 仍於發送前執行。 |
| **高風險/法務/詐騙短路** | 未改。legal_risk 與安全確認模板之觸發與 return 不變。 |
| **尷尬/重複轉人工** | 未改。shouldHandoffDueToAwkwardOrRepeat 與呼叫時機不變。 |
| **圖片純圖固定模板** | 未改。webhook 純圖片路徑與 vision-first 行為不變。 |

## 二、哪些只是表現層調整

僅改 **system prompt 的用語與語氣**，不新增/刪除任何 if/else 或短路：

| 位置 | 調整內容 |
|------|----------|
| **getEnrichedSystemPrompt（新增 toneBlock）** | 在 base + brand 之後、handoffBlock 之前，加入「回覆風格」總則：像一般對話、自然接話、減少問卷感與公告感；底線由下方規則與發送前 guard 把關。 |
| **handoffBlock 內「二、詢問訂單／出貨進度」** | 規則不變（只允許兩種方式、不得問其他欄位、查不到轉人工、不講平台）。用語改為「用一兩句自然承接即可」「不要列成選單或一次丟很多欄位」。 |
| **handoffBlock 內「五、語氣與禁止」** | 保留全域禁止平台話術。補「像真人對話即可：自然短句、口語承接，不必制式開場或列點問卷」，並註明發送前也會檢查。 |
| **本輪 handoff 附加指引** | 規則不變（呼叫 transfer_to_human、明講轉真人、不補問訂單/產品+手機）。例句改為「好喔我幫您轉真人客服處理～」等較口語說法。 |
| **本輪 off_topic_guard** | 規則不變（不收錄推薦、不展開）。補「像朋友隨口帶過」「不要像公告」、約 30～50 字。 |
| **本輪 查單** | 規則不變（只接受訂單編號或產品+手機、不得問其他欄位、不得重問）。補「用一兩句自然承接後再引導其中一種即可，不要列成選單或問卷」。 |
| **本輪 久候型售後 / 商品問題型退換貨** | 邏輯不變。用語改為「先一兩句自然道歉與承接」「先一句自然安撫（如不好意思久等～）」等短句描述。 |
| **本輪 客戶目標鎖定** | 改為「本輪重點」、「不要岔到…」等口語化表述，實質約束不變。 |
| **本輪禁止（no-promo）** | 禁止項目不變，僅略為精簡用字（如「不介紹產品、不推銷」→「不推銷」）。 |

本輪**未再縮小或移除**任何固定模板的觸發條件；先前已做過的調整維持不變：

| 項目 | 現況 |
|------|------|
| **圖片＋短句固定模板** | 已於第一階段收斂移除（有文字一律進主流程），本輪未動。 |
| **off_topic 固定句** | 已於第一階段改為由模型收邊界，本輪未動。 |
| **安全確認模板** | 觸發門檻已提高（safe_confirm_order 須有來源不明意圖），本輪未動。 |

## 五、怎麼確保這次不會跟以前的規則打架

1. **只改 prompt 字串**  
   所有變更都在 `getEnrichedSystemPrompt` 與 `systemPrompt += ...` 的**內容**上，沒有改 state、plan、分支條件、工具回傳格式或 guard 的呼叫時機與邏輯。

2. **底線條文仍寫在 prompt 裡**  
   「查單只接受兩種方式」「不得問其他欄位」「不得提及其他平台／官方通路」「轉人工要明講」「午休/下班要明講」等仍清楚寫在 handoffBlock 與各 mode 指引中，模型仍被約束，只是用語從「步驟/問卷」改為「自然承接＋底線」。

3. **發送前 guard 未動**  
   即便模型偶爾用詞擦邊，platform / official_channel / no-promo / category / output 等 guard 仍會在發送前攔截或替換，底線由程式保證。

4. **未新增任何分支**  
   沒有新增 if/else、沒有新短路、沒有新 template；僅在既有 flow 下讓模型被引導成「更像對話、少像問卷」。

## 六、Before / After 範例（證明變自然、底線沒破）

以下為**預期風格**對照，實際輸出仍會隨模型與情境略有差異；底線行為（只引導兩種查單方式、不講平台、轉人工明講、午休明講、guard 擋禁語）維持不變。

---

### 一般招呼

| Before（偏規則/問卷感） | After（表現層目標） |
|-------------------------|----------------------|
| 「您好，請問您今天需要什麼協助呢？您可以選擇：1. 訂單查詢 2. 商品問題 3. 退換貨…」 | 「嗨～有什麼需要幫您的嗎？」或「您好，想問什麼跟我說～」 |

---

### 查單引導（底線：仍只接受訂單編號或產品+手機，不問其他欄位）

| Before（偏問卷感） | After（表現層目標） |
|--------------------|----------------------|
| 「了解，這邊先幫您確認一下出貨狀況。請提供以下其中一項：① 訂單編號 ② 產品名稱＋手機號碼。」 | 「好喔我幫您查一下～方便給我訂單編號嗎？」或「不好意思久等～有訂單編號或商品名+手機的話跟我說，我幫您查。」 |

---

### 品牌外話題收邊界（底線：不推薦、不展開）

| Before（偏公告感） | After（表現層目標） |
|--------------------|----------------------|
| 「這部分比較不是我們服務範圍～若有商品或訂單相關問題，可以跟我說，我幫您處理😊」 | 「這塊我們比較沒辦法幫上忙～有商品或訂單問題再跟我說喔」 |

---

### 轉人工（底線：必須明講已轉真人；午休/下班必須明講之後才會回）

| Before（可接受） | After（表現層目標，底線不變） |
|------------------|-------------------------------|
| 「了解，我先幫您整理目前狀況，接著安排真人客服為您接手處理，這樣會比較快一些🙏」 | 「好喔我幫您轉真人客服處理～」或「我這邊幫您轉真人專員，請稍候～」 |
| 午休/下班時仍由程式強制補句：「目前人工客服不在線，之後才會由真人客服回覆。」 | 同上，補句邏輯未改，仍為程式底線。 |

---

### 久候型售後（底線：先安撫、先查詢、不先表單、不先轉人工）

| Before（可接受） | After（表現層目標） |
|------------------|----------------------|
| 「不好意思讓您久等了。我先幫您查詢出貨狀況，說明是否有現貨、能否加急…」 | 「不好意思久等～我幫您看一下目前出貨狀況，有現貨會盡快幫您安排，也會幫您備註加急」 |

---

### 底線檢查（不應出現）

- 查單時仍**只**引導「訂單編號」或「產品+手機」，**不會**出現「請提供購買頁面／收件資料／官方通路」等。
- 回覆中**不會**出現「其他平台」「該平台」「官方通路」「建議向該平台客服確認」等（由 content guard 擋）。
- 轉人工時**一定會**有「轉接真人專員／轉真人客服」等明講（由程式固定句或 prompt 約束）。
- 午休/下班轉人工時**一定會**有「目前人工客服不在線，之後才會由真人客服回覆」等（由程式補句）。

---

以上為本次「表現層鬆綁」之完成報告：核心邏輯與 guard 全數保留，僅在 system prompt 的表現層做語氣與風格調整，以降低規則感、提升像真人的對話感。
